Web App Manifest Favicon Guide: Master Mobile Icons in 2026

If you have ever added a website to your Android home screen only to see a tiny icon trapped inside a giant, ugly white circle, you have witnessed a manifest failure. While most developers focus heavily on the apple-touch-icon for iOS users, the Android ecosystem relies on a much more structured approach: the Web App Manifest. In 2026, simply having a 16x16 ICO file is not enough to look professional on the billions of devices running Chrome and other Chromium-based browsers.

A web app manifest favicon is not technically a single file, but a declaration within a JSON file that tells the browser how your site should behave when installed as an app. This includes the icon, the theme color, and even the splash screen that appears while your site loads. If you want your site to feel like a first-class citizen on mobile, you need to get this right.

The Core of the Web App Manifest

The manifest is a simple JSON file, usually named manifest.json or site.webmanifest. It lives in your root directory and acts as a central registry for your site's metadata. Unlike standard favicons that are linked directly in the HTML, the manifest allows you to provide a variety of icon sizes and purposes in a single block of code.

Google and the Chromium team have standardized this to ensure that Progressive Web Apps (PWAs) and even standard websites look consistent. When a user clicks "Add to Home Screen," the browser looks for this file to determine which image to use. If it is missing, Chrome will often generate a generic icon based on the first letter of your domain or a low-resolution screenshot of the page—both of which look terrible.

Prerequisites for a Modern Setup

Before we touch the code, you need high-quality source images. We recommend starting with a 512x512 pixel PNG. This size is large enough to cover the highest-density mobile displays and the PWA splash screens. You should also have a version of your icon that works well as a "maskable" icon (more on that in a moment). If you are starting from scratch, you might want to read our guide on favicon best practices in 2026 to ensure your design is ready for scaling.

Step 1: Creating the manifest.json File

Create a file named manifest.json in your project root. This file needs to be served with the application/manifest+json MIME type, though most modern servers handle this automatically if you use the .json extension. Here is a baseline configuration focused on icons:

{
  "name": "Mzu favicondl Pro Tool",
  "short_name": "Mzu",
  "icons": [
    {
      "src": "/icons/icon-192x192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/icons/icon-512x512.png",
      "sizes": "512x512",
      "type": "image/png"
    }
  ],
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#000000"
}

The icons array is where the magic happens. The 192x192 icon is typically used for the home screen, while the 512x512 icon is used for the splash screen that appears when the app is launching. We have seen many developers skip the 512px version, resulting in a blurry splash screen on high-end devices like the latest Pixel or Galaxy phones.

Step 2: The Secret Ingredient — Maskable Icons

This is where most people get it wrong. Android uses different shapes for icons: circles, squares, rounded rectangles, or even "squiurcles." If your icon is a simple square PNG with a transparent background, Android will shrink it and place it inside a white shape to ensure it fits the system theme. This looks amateur.

To fix this, you need a maskable icon. This is an icon where the artwork is contained within a "safe zone" (the inner 80% of the image), and the background extends to the edges. You then tell the browser that this icon is safe to crop by adding "purpose": "maskable" to the icon object.

{
  "src": "/icons/maskable-icon-512x512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "maskable"
}

By providing both a standard icon and a maskable one, you give the OS the flexibility to display your brand perfectly regardless of the user's UI settings. (Pro tip: You can actually combine them by using "purpose": "any maskable", but we prefer separate files for better control over padding).

Step 3: Linking the Manifest in HTML

Once your JSON file and images are ready, you must tell the browser where to find them. Add the following line to the <head> of your HTML documents:

<link rel="manifest" href="/manifest.json">

Wait, is that it? Not quite. While the manifest handles Android and PWAs, you still need the apple-touch-icon for Safari. For a deep dive on that specific requirement, check out our Apple Touch Icon complete setup guide. The two systems work together to cover the entire mobile market.

Step 4: Testing and Validation

Don't just assume it works because the code looks clean. Open your site in Chrome, hit F12 to open DevTools, and go to the Application tab. On the left sidebar, click Manifest. Chrome will tell you if the manifest was found, if the icons are the correct size, and even show you a preview of the maskable icon safe zone.

Common Pitfalls to Avoid

One of the most frequent issues we see at Mzu favicondl is the caching problem. Browsers are incredibly aggressive about caching the manifest.json file. If you update your icons but don't change the filenames or the manifest content, users might not see the update for weeks. Always use versioning (e.g., icon-192x192.png?v=2) if you are pushing a brand refresh.

Another mistake is neglecting the theme_color. This property doesn't just affect the manifest; it tells the mobile browser what color to paint the address bar and the system status bar. If your icon is dark blue but your theme_color is white, the transition from the home screen to your app will feel jarring and unpolished.

Finally, ensure your paths are correct. Relative paths in a manifest file are relative to the manifest file's location, not the HTML page. We always recommend using root-relative paths (starting with /) to avoid confusion, especially on sites with deep URL structures.

Setting up a web app manifest favicon might feel like extra homework compared to the old days of dropping a favicon.ico in the root. However, the payoff is a professional, native-feeling presence on the world's most popular mobile operating system. Take the ten minutes to do it right—your users will notice the difference, even if they can't quite put their finger on why your site looks so much better than the competition.

第二步：核心秘诀 —— 可遮罩图标 (Maskable Icons)

这是大多数人出错的地方。安卓系统使用不同的图标形状：圆形、正方形、圆角矩形，甚至是“超椭圆”。如果你的图标只是一个带透明背景的普通正方形 PNG，安卓会缩小它并将其放入一个白色形状中，以匹配系统主题。这看起来非常不专业。

要解决这个问题，你需要一个可遮罩图标。这种图标的艺术内容包含在“安全区域”（图像内部 80% 的圆圈区域）内，背景延伸到边缘。然后你通过在图标对象中添加 "purpose": "maskable" 来告诉浏览器这个图标可以被安全裁剪。

{
  "src": "/icons/maskable-icon-512x512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "maskable"
}

通过同时提供标准图标和可遮罩图标，你可以让操作系统根据用户的 UI 设置完美地展示你的品牌。（专业提示：你其实可以使用 "purpose": "any maskable" 将两者合二为一，但我们更倾向于使用独立文件，以便更好地控制边距）。

Web App Manifestの核となる部分

マニフェストは、通常manifest.jsonまたはsite.webmanifestという名前のシンプルなJSONファイルです。ルートディレクトリに配置され、サイトのメタデータのための中央レジストリとして機能します。HTMLで直接リンクされる標準のfaviconとは異なり、マニフェストでは様々なサイズのアイコンとその目的を単一のコードブロックで提供できます。

GoogleとChromiumチームは、Progressive Web Apps (PWA)や標準的なウェブサイトでさえも一貫した見た目になるように、これを標準化しました。ユーザーが「ホーム画面に追加」をクリックすると、ブラウザはこのファイルを検索して使用する画像を決定します。このファイルが見つからない場合、Chromeはドメインの最初の文字やページの低解像度スクリーンショットに基づいて汎用アイコンを生成することが多く、どちらも見栄えが非常に悪くなります。

ステップ2: 成功の秘訣 — Maskable Icons

ここが、ほとんどの人が間違える点です。Androidはアイコンに様々な形を使用します。円、四角、角丸四角形、あるいは「squircle」と呼ばれる形などです。もしあなたのアイコンが透明な背景を持つ単純な四角形のPNGである場合、Androidはそれを縮小し、システムテーマに合うように白い図形の中に配置します。これでは素人っぽく見えてしまいます。

これを解決するには、maskable iconが必要です。これは、アートワークが「セーフゾーン」（画像の内部80%）に収められ、背景が端まで広がっているアイコンです。そして、アイコンオブジェクトに"purpose": "maskable"を追加することで、このアイコンが切り抜かれても安全であることをブラウザに伝えます。

{
  "src": "/icons/maskable-icon-512x512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "maskable"
}

標準アイコンとmaskableアイコンの両方を提供することで、ユーザーのUI設定に関わらず、あなたのブランドを完璧に表示する柔軟性をOSに与えることができます。（プロのヒント：実際には"purpose": "any maskable"を使用してそれらを組み合わせることもできますが、パディングをより細かく制御するために、私たちはファイルを分けることを好みます）。

避けるべき一般的な落とし穴

Mzu favicondlで私たちが最も頻繁に目にする問題の1つは、キャッシュの問題です。ブラウザはmanifest.jsonファイルを非常に積極的にキャッシュします。アイコンを更新しても、ファイル名やマニフェストの内容を変更しないと、ユーザーは何週間も更新に気づかないかもしれません。ブランドのリフレッシュをプッシュする場合は、常にバージョニング（例：icon-192x192.png?v=2）を使用してください。

最後に、パスが正しいことを確認してください。マニフェストファイル内の相対パスは、HTMLページからではなく、マニフェストファイルの場所を基準とします。特に深いURL構造を持つサイトでは、混乱を避けるために常にルート相対パス（/で始まる）を使用することをお勧めします。

Web App Manifestのfaviconを設定することは、ルートにfavicon.icoを置くだけだった昔と比べると、少し手間がかかるように感じるかもしれません。しかし、その見返りは、世界で最も人気のあるモバイルオペレーティングシステム上でのプロフェッショナルで、ネイティブアプリのような存在感です。10分かけて正しく設定しましょう。ユーザーは、なぜあなたのサイトが競合よりもずっと良く見えるのか、その理由を正確には説明できなくても、その違いに気づくはずです。

Web App Manifest favicon은 기술적으로 단일 파일이 아니라, 사이트가 앱으로 설치될 때 브라우저에 어떻게 동작해야 하는지를 알려주는 JSON 파일 내의 선언입니다. 여기에는 아이콘, 테마 색상, 심지어 사이트가 로드되는 동안 나타나는 스플래시 화면까지 포함됩니다. 당신의 사이트가 모바일에서 네이티브 앱과 같은 일류 경험을 제공하기를 원한다면, 이 설정을 제대로 해야 합니다.

Web App Manifest의 핵심

매니페스트는 보통 manifest.json 또는 site.webmanifest라는 이름의 간단한 JSON 파일입니다. 프로젝트의 루트 디렉토리에 위치하며 사이트의 메타데이터를 위한 중앙 레지스트리 역할을 합니다. HTML에서 직접 링크되는 표준 favicon과 달리, 매니페스트를 사용하면 단일 코드 블록에서 다양한 아이콘 크기와 목적을 제공할 수 있습니다.

Google과 Chromium 팀은 Progressive Web Apps(PWA)와 심지어 표준 웹사이트까지 일관성 있게 보이도록 이 방식을 표준화했습니다. 사용자가 '홈 화면에 추가'를 클릭하면 브라우저는 이 파일을 찾아 사용할 이미지를 결정합니다. 이 파일이 없으면 Chrome은 종종 도메인의 첫 글자나 페이지의 저해상도 스크린샷을 기반으로 일반 아이콘을 생성하는데, 두 경우 모두 끔찍하게 보입니다.

최신 설정을 위한 전제 조건

코드를 만지기 전에 고품질의 원본 이미지가 필요합니다. 512x512 픽셀 PNG로 시작하는 것을 권장합니다. 이 크기는 최고 밀도의 모바일 디스플레이와 PWA 스플래시 화면을 모두 커버할 수 있을 만큼 충분히 큽니다. 또한 'maskable' 아이콘(잠시 후에 자세히 설명)으로 잘 작동하는 버전의 아이콘도 있어야 합니다. 처음부터 시작하는 경우, 2026년 favicon 모범 사례에 대한 저희 가이드를 읽고 디자인이 확장에 준비되었는지 확인하는 것이 좋습니다.

1단계: manifest.json 파일 생성하기

프로젝트 루트에 manifest.json이라는 이름의 파일을 만드세요. 이 파일은 application/manifest+json MIME 타입으로 제공되어야 하지만, 대부분의 최신 서버는 .json 확장자를 사용하면 이를 자동으로 처리합니다. 다음은 아이콘에 초점을 맞춘 기본 구성입니다.

{
  "name": "Mzu favicondl Pro Tool",
  "short_name": "Mzu",
  "icons": [
    {
      "src": "/icons/icon-192x192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/icons/icon-512x512.png",
      "sizes": "512x512",
      "type": "image/png"
    }
  ],
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#000000"
}

icons 배열이 바로 마법이 일어나는 곳입니다. 192x192 아이콘은 일반적으로 홈 화면에 사용되며, 512x512 아이콘은 앱이 실행될 때 나타나는 스플래시 화면에 사용됩니다. 많은 개발자들이 512px 버전을 건너뛰는 것을 보았는데, 이는 최신 Pixel이나 Galaxy 폰과 같은 고급 기기에서 스플래시 화면이 흐릿하게 보이는 결과를 낳습니다.

2단계: 비밀 병기 — Maskable Icons

이 부분이 대부분의 사람들이 잘못하는 지점입니다. Android는 원, 사각형, 둥근 사각형, 심지어 '스쿼클(squircle)'과 같은 다양한 모양의 아이콘을 사용합니다. 만약 당신의 아이콘이 투명한 배경을 가진 단순한 사각형 PNG라면, Android는 시스템 테마에 맞추기 위해 아이콘을 축소하여 흰색 도형 안에 배치할 것입니다. 이것은 아마추어처럼 보입니다.

이 문제를 해결하려면 maskable icon이 필요합니다. 이것은 아트워크가 '안전 영역'(이미지의 내부 80%) 내에 포함되고 배경이 가장자리까지 확장되는 아이콘입니다. 그런 다음 아이콘 객체에 "purpose": "maskable"을 추가하여 이 아이콘을 잘라내도 안전하다고 브라우저에 알려줍니다.

{
  "src": "/icons/maskable-icon-512x512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "maskable"
}

표준 아이콘과 maskable 아이콘을 모두 제공함으로써, 사용자의 UI 설정에 관계없이 OS가 당신의 브랜드를 완벽하게 표시할 수 있는 유연성을 부여하게 됩니다. (프로 팁: 실제로는 "purpose": "any maskable"을 사용하여 둘을 결합할 수도 있지만, 저희는 패딩을 더 잘 제어하기 위해 파일을 분리하는 것을 선호합니다).

3단계: HTML에서 매니페스트 링크하기

JSON 파일과 이미지가 준비되면, 브라우저에 그것들을 어디서 찾을 수 있는지 알려줘야 합니다. HTML 문서의 <head>에 다음 줄을 추가하세요.

<link rel="manifest" href="/manifest.json">

잠깐, 이게 다인가요? 그렇지는 않습니다. 매니페스트가 Android와 PWA를 처리하는 동안, Safari를 위해서는 여전히 apple-touch-icon이 필요합니다. 해당 특정 요구 사항에 대한 심층적인 내용은 Apple Touch Icon 완벽 설정 가이드를 확인하세요. 두 시스템은 함께 작동하여 전체 모바일 시장을 커버합니다.

4단계: 테스트 및 검증

코드가 깔끔해 보인다고 해서 작동한다고 가정하지 마세요. Chrome에서 사이트를 열고 F12를 눌러 개발자 도구를 연 다음, Application 탭으로 이동하세요. 왼쪽 사이드바에서 Manifest를 클릭하세요. Chrome은 매니페스트를 찾았는지, 아이콘 크기가 올바른지 알려주고, 심지어 maskable 아이콘의 안전 영역 미리보기도 보여줄 것입니다.

피해야 할 일반적인 함정

Mzu favicondl에서 우리가 가장 자주 보는 문제 중 하나는 캐싱 문제입니다. 브라우저는 manifest.json 파일을 믿을 수 없을 정도로 공격적으로 캐싱합니다. 아이콘을 업데이트했지만 파일 이름이나 매니페스트 내용을 변경하지 않으면, 사용자들은 몇 주 동안 업데이트를 보지 못할 수도 있습니다. 브랜드 리프레시를 적용할 때는 항상 버전 관리(예: icon-192x192.png?v=2)를 사용하세요.

또 다른 실수는 theme_color를 소홀히 하는 것입니다. 이 속성은 매니페스트에만 영향을 미치는 것이 아니라, 모바일 브라우저에 주소 표시줄과 시스템 상태 표시줄을 어떤 색으로 칠할지 알려줍니다. 아이콘이 진한 파란색인데 theme_color가 흰색이라면, 홈 화면에서 앱으로의 전환이 어색하고 세련되지 않게 느껴질 것입니다.

마지막으로, 경로가 올바른지 확인하세요. 매니페스트 파일의 상대 경로는 HTML 페이지가 아닌 매니페스트 파일의 위치를 기준으로 합니다. 특히 깊은 URL 구조를 가진 사이트에서는 혼동을 피하기 위해 항상 루트 상대 경로(/로 시작)를 사용하는 것을 권장합니다.

Web App Manifest favicon을 설정하는 것은 루트에 favicon.ico 파일을 던져 넣던 옛날에 비하면 추가적인 숙제처럼 느껴질 수 있습니다. 하지만 그 보상은 세계에서 가장 인기 있는 모바일 운영 체제에서 전문적이고 네이티브 앱처럼 느껴지는 존재감을 얻는 것입니다. 10분만 투자해서 제대로 설정하세요. 사용자들은 왜 당신의 사이트가 경쟁사보다 훨씬 좋아 보이는지 정확히 집어내지는 못하더라도 그 차이를 알아차릴 것입니다.

Si alguna vez has añadido un sitio web a tu pantalla de inicio de Android y has visto un diminuto icono atrapado dentro de un círculo blanco gigante y feo, has sido testigo de un fallo del manifiesto. Aunque la mayoría de los desarrolladores se centran mucho en el apple-touch-icon para los usuarios de iOS, el ecosistema de Android se basa en un enfoque mucho más estructurado: el Web App Manifest. En 2026, tener simplemente un archivo ICO de 16x16 no es suficiente para parecer profesional en los miles de millones de dispositivos que ejecutan Chrome y otros navegadores basados en Chromium.

Un favicon de manifiesto de aplicación web no es técnicamente un único archivo, sino una declaración dentro de un archivo JSON que le dice al navegador cómo debe comportarse tu sitio cuando se instala como una aplicación. Esto incluye el icono, el color del tema e incluso la pantalla de bienvenida (splash screen) que aparece mientras se carga tu sitio. Si quieres que tu sitio se sienta como una aplicación nativa de primer nivel en el móvil, necesitas hacer esto bien.

El núcleo del Web App Manifest

El manifiesto es un simple archivo JSON, normalmente llamado manifest.json o site.webmanifest. Se encuentra en tu directorio raíz y actúa como un registro central para los metadatos de tu sitio. A diferencia de los favicons estándar que se enlazan directamente en el HTML, el manifiesto te permite proporcionar una variedad de tamaños y propósitos de iconos en un solo bloque de código.

Google y el equipo de Chromium han estandarizado esto para asegurar que las Progressive Web Apps (PWAs) e incluso los sitios web estándar tengan una apariencia consistente. Cuando un usuario hace clic en "Añadir a la pantalla de inicio", el navegador busca este archivo para determinar qué imagen usar. Si falta, Chrome a menudo generará un icono genérico basado en la primera letra de tu dominio o una captura de pantalla de baja resolución de la página, y ambas opciones se ven terribles.

Requisitos previos para una configuración moderna

Antes de que toquemos el código, necesitas imágenes fuente de alta calidad. Recomendamos empezar con un PNG de 512x512 píxeles. Este tamaño es lo suficientemente grande como para cubrir las pantallas móviles de mayor densidad y las pantallas de bienvenida de las PWA. También deberías tener una versión de tu icono que funcione bien como un icono "maskable" (más sobre esto en un momento). Si estás empezando desde cero, quizás quieras leer nuestra guía sobre las mejores prácticas para favicons en 2026 para asegurarte de que tu diseño está listo para escalar.

Paso 1: Crear el archivo manifest.json

Crea un archivo llamado manifest.json en la raíz de tu proyecto. Este archivo debe servirse con el tipo MIME application/manifest+json, aunque la mayoría de los servidores modernos lo manejan automáticamente si usas la extensión .json. Aquí tienes una configuración básica centrada en los iconos:

{
  "name": "Mzu favicondl Pro Tool",
  "short_name": "Mzu",
  "icons": [
    {
      "src": "/icons/icon-192x192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/icons/icon-512x512.png",
      "sizes": "512x512",
      "type": "image/png"
    }
  ],
  "start_url": "/",
  "display": "standalone",
  "background_color": "#ffffff",
  "theme_color": "#000000"
}

El array icons es donde ocurre la magia. El icono de 192x192 se usa típicamente para la pantalla de inicio, mientras que el icono de 512x512 se usa para la pantalla de bienvenida que aparece cuando la aplicación se está lanzando. Hemos visto a muchos desarrolladores omitir la versión de 512px, lo que resulta en una pantalla de bienvenida borrosa en dispositivos de gama alta como los últimos teléfonos Pixel o Galaxy.

Paso 2: El ingrediente secreto — Iconos adaptables (Maskable Icons)

Aquí es donde la mayoría de la gente se equivoca. Android usa diferentes formas para los iconos: círculos, cuadrados, rectángulos redondeados o incluso "squiurcles". Si tu icono es un simple PNG cuadrado con un fondo transparente, Android lo encogerá y lo colocará dentro de una forma blanca para asegurarse de que se ajuste al tema del sistema. Esto parece de aficionado.

Para solucionar esto, necesitas un icono adaptable (maskable icon). Este es un icono donde el diseño está contenido dentro de una "zona segura" (el 80% interior de la imagen), y el fondo se extiende hasta los bordes. Luego le dices al navegador que este icono es seguro para recortar añadiendo "purpose": "maskable" al objeto del icono.

{
  "src": "/icons/maskable-icon-512x512.png",
  "sizes": "512x512",
  "type": "image/png",
  "purpose": "maskable"
}

Al proporcionar tanto un icono estándar como uno adaptable, le das al sistema operativo la flexibilidad de mostrar tu marca perfectamente sin importar la configuración de la interfaz de usuario del usuario. (Consejo profesional: en realidad puedes combinarlos usando "purpose": "any maskable", pero preferimos archivos separados para un mejor control sobre el relleno).

Paso 3: Enlazar el manifiesto en HTML

Una vez que tu archivo JSON y tus imágenes estén listos, debes decirle al navegador dónde encontrarlos. Añade la siguiente línea en el <head> de tus documentos HTML:

<link rel="manifest" href="/manifest.json">

Espera, ¿eso es todo? No exactamente. Aunque el manifiesto se encarga de Android y las PWAs, todavía necesitas el apple-touch-icon para Safari. Para una inmersión profunda en ese requisito específico, echa un vistazo a nuestra guía completa de configuración del Apple Touch Icon. Los dos sistemas trabajan juntos para cubrir todo el mercado móvil.

Paso 4: Pruebas y validación

No asumas que funciona solo porque el código se ve limpio. Abre tu sitio en Chrome, presiona F12 para abrir las DevTools y ve a la pestaña Application. En la barra lateral izquierda, haz clic en Manifest. Chrome te dirá si se encontró el manifiesto, si los iconos tienen el tamaño correcto e incluso te mostrará una vista previa de la zona segura del icono adaptable.

Errores comunes que debes evitar

Uno de los problemas más frecuentes que vemos en Mzu favicondl es el problema de caché. Los navegadores son increíblemente agresivos al almacenar en caché el archivo manifest.json. Si actualizas tus iconos pero no cambias los nombres de los archivos o el contenido del manifiesto, es posible que los usuarios no vean la actualización durante semanas. Siempre usa versionado (p. ej., icon-192x192.png?v=2) si estás lanzando una actualización de marca.

Otro error es descuidar el theme_color. Esta propiedad no solo afecta al manifiesto; le dice al navegador móvil de qué color pintar la barra de direcciones y la barra de estado del sistema. Si tu icono es azul oscuro pero tu theme_color es blanco, la transición desde la pantalla de inicio a tu aplicación se sentirá discordante y poco pulida.

Finalmente, asegúrate de que tus rutas sean correctas. Las rutas relativas en un archivo de manifiesto son relativas a la ubicación del archivo de manifiesto, no a la página HTML. Siempre recomendamos usar rutas relativas a la raíz (que comienzan con /) para evitar confusiones, especialmente en sitios con estructuras de URL profundas.

Configurar un favicon de manifiesto de aplicación web puede parecer un trabajo adicional en comparación con los viejos tiempos de simplemente soltar un favicon.ico en la raíz. Sin embargo, la recompensa es una presencia profesional y con una sensación nativa en el sistema operativo móvil más popular del mundo. Tómate los diez minutos para hacerlo bien; tus usuarios notarán la diferencia, incluso si no pueden explicar exactamente por qué tu sitio se ve mucho mejor que el de la competencia.
