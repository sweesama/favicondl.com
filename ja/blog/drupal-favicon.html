Drupal Favicon Setup Guide: Stop Serving the Blue Drop

If you've ever launched a custom Drupal 10 or 11 site, you know the drill. You spend weeks perfecting the Twig templates, configuring complex Views, and mapping out taxonomy structures. Then, you open Safari to admire your work, and that stubborn blue Drupal water drop is still staring back at you from the tab bar.

Getting rid of that default core icon requires more than just dropping an image into your public directory. Drupal's aggressive caching and theme inheritance make this seemingly simple task surprisingly frustrating. In this drupal favicon setup guide, we will walk through the exact steps to replace the core icon permanently, ensuring your brand looks crisp across all devices in 2026.

Look at how large enterprise teams (like Red Hat or major universities) handle their Drupal platforms. They rarely rely on the admin UI for critical branding assets. Instead, they hardcode a modern SVG directly into their theme layer. We are going to take that exact professional approach.

Why the Default UI Method Fails

Drupal does offer a built-in way to change the favicon. If you navigate to Appearance > Settings > Your Theme, you will find a toggle to uncheck the default shortcut icon and upload a custom one.

I strongly recommend against using this method for production sites.

Why? First, it relies on the managed file system, meaning your icon URL will look something like /sites/default/files/favicon.ico. If you ever migrate databases or lose your files directory, your icon breaks. Second, the UI only accepts a single legacy ICO file, completely ignoring the modern 2026 stack of SVGs and Apple Touch Icons.

The Professional Drupal Favicon Setup Guide

To do this right, we need to bypass the UI and inject our icons directly into the theme. Before we touch any code, generate your modern icon set using Mzu favicondl. You want a crisp SVG and a fallback ICO.

Step 1: Place Files in Your Theme

Instead of the public files directory, store your icons inside your custom theme. Create a dedicated folder for them to keep things organized and version-controlled.

  • Navigate to /web/themes/custom/your_theme/
  • Create a directory named icons
  • Drop your favicon.svg and favicon.ico into this folder.

Step 2: Override html.html.twig

Drupal core controls the <head> output via the html.html.twig template. If your theme doesn't already have this file, copy it from the core system module (/core/modules/system/templates/html.html.twig) into your theme's /templates/layout/ directory.

Open the file. You will see a <head-placeholder token=...> tag. Just below that, hardcode your modern SVG favicon links using Twig's active theme path variable.

<link rel='icon' href='{{ active_theme_path() }}/icons/favicon.svg' type='image/svg+xml'>
<link rel='alternate icon' href='{{ active_theme_path() }}/icons/favicon.ico'>
<link rel='apple-touch-icon' href='{{ active_theme_path() }}/icons/apple-touch-icon.png'>

This guarantees your icons are deployed with your codebase and never rely on the database state.

Step 3: Remove the Core Attachment

Even with your hardcoded Twig tags, Drupal might still inject its default icon tag via page attachments, causing duplicate tags in your DOM. To strip it out, we use a preprocess hook in your your_theme.theme file.

/**
 * Implements hook_page_attachments_alter().
 */
function your_theme_page_attachments_alter(array &$attachments) {
  foreach ($attachments['#attached']['html_head_link'] as $key => $value) {
    if (isset($value[0]['rel']) && $value[0]['rel'] === 'icon') {
      unset($attachments['#attached']['html_head_link'][$key]);
    }
  }
}

This snippet intercepts the page build process and aggressively strips out any icon links Drupal tries to inject natively, leaving only your clean Twig implementation.

Busting the Drupal Cache

You can make all the code changes in the world, but Drupal won't show them until you clear the cache. The render cache is notoriously stubborn.

  1. Open your terminal and navigate to your project root.
  2. Run drush cr (Cache Rebuild).
  3. If you are using a reverse proxy like Varnish or a CDN like Cloudflare, purge those caches as well.

If you still see the old blue drop, your browser is likely holding onto it. Read our guide on forcing a favicon cache clear to resolve local browser stubbornness.

Common Pitfalls in Drupal 10/11

If your setup is still failing, check these common culprits:

  • File Permissions: Ensure your /icons/ directory is readable by the web server (usually 755 for directories, 644 for files).
  • Base Path Issues: If you are running Drupal in a subdirectory, active_theme_path() might need to be prefixed with the base URL.
  • Theme Inheritance: If you are building a sub-theme (e.g., based on Radix or Olivero), make sure your parent theme isn't injecting its own icons via a higher-priority hook.

  foreach ($attachments['#attached']['html_head_link'] as $key => $value) {
    if (isset($value[0]['rel']) && $value[0]['rel'] === 'icon') {
      unset($attachments['#attached']['html_head_link'][$key]);
    }
  }
}

이 스니펫은 페이지 빌드 프로세스를 가로채어 Drupal이 기본적으로 주입하려는 모든 아이콘 링크를 강제로 제거하고 깔끔한 Twig 구현만 남깁니다.

Drupal 캐시 지우기

코드를 아무리 많이 변경하더라도 캐시를 지울 때까지 Drupal은 이를 표시하지 않습니다. 렌더링 캐시는 매우 고집이 셉니다.

  1. 터미널을 열고 프로젝트 루트로 이동합니다.
  2. drush cr(Cache Rebuild)을 실행합니다.
  3. Varnish와 같은 리버스 프록시나 Cloudflare와 같은 CDN을 사용하는 경우 해당 캐시도 제거하세요.

여전히 예전의 파란 물방울이 보인다면 브라우저가 이를 붙잡고 있을 가능성이 높습니다. 로컬 브라우저의 고집을 해결하려면 파비콘 캐시 강제 지우기 가이드를 읽어보세요.

Drupal 10/11의 일반적인 함정

설정이 여전히 실패하는 경우 다음 일반적인 원인을 확인하세요.

  • 파일 권한: /icons/ 디렉토리를 웹 서버에서 읽을 수 있는지 확인합니다(일반적으로 디렉토리는 755, 파일은 644).
  • 기본 경로 문제: 하위 디렉토리에서 Drupal을 실행하는 경우 active_theme_path()에 기본 URL 접두사가 필요할 수 있습니다.
  • 테마 상속: 하위 테마(예: Radix 또는 Olivero 기반)를 구축하는 경우 상위 테마가 더 높은 우선순위의 후크를 통해 자체 아이콘을 주입하지 않는지 확인하세요.

Por qué falla el método de la UI por defecto

Drupal ofrece una forma integrada de cambiar el favicon. Si navegas a Appearance > Settings > Your Theme, encontrarás un interruptor para desmarcar el icono de acceso directo por defecto y subir uno personalizado.

Recomiendo encarecidamente no usar este método para sitios en producción.

¿Por qué? Primero, depende del sistema de archivos administrado, lo que significa que la URL de tu icono se verá como /sites/default/files/favicon.ico. Si alguna vez migras bases de datos o pierdes tu directorio de archivos, tu icono se rompe. Segundo, la UI solo acepta un único archivo ICO heredado, ignorando por completo la pila moderna de 2026 de SVGs y Apple Touch Icons.

La Guía Profesional de Configuración de Favicon en Drupal

Para hacer esto bien, necesitamos evitar la UI e inyectar nuestros iconos directamente en el tema. Antes de tocar ningún código, genera tu conjunto de iconos modernos usando Mzu favicondl. Necesitas un SVG nítido y un ICO de respaldo.

Paso 1: Coloca los archivos en tu tema

En lugar del directorio de archivos públicos, almacena tus iconos dentro de tu tema personalizado. Crea una carpeta dedicada para ellos para mantener las cosas organizadas y bajo control de versiones.

  • Navega a /web/themes/custom/your_theme/
  • Crea un directorio llamado icons
  • Suelta tu favicon.svg y favicon.ico en esta carpeta.

Paso 2: Sobrescribe html.html.twig

El núcleo de Drupal controla la salida del <head> a través de la plantilla html.html.twig. Si tu tema aún no tiene este archivo, cópialo desde el módulo del sistema central (/core/modules/system/templates/html.html.twig) al directorio /templates/layout/ de tu tema.

Abre el archivo. Verás una etiqueta <head-placeholder token=...>. Justo debajo de eso, codifica tus enlaces de favicon SVG moderno usando la variable de ruta del tema activo de Twig.

<link rel='icon' href='{{ active_theme_path() }}/icons/favicon.svg' type='image/svg+xml'>
<link rel='alternate icon' href='{{ active_theme_path() }}/icons/favicon.ico'>
<link rel='apple-touch-icon' href='{{ active_theme_path() }}/icons/apple-touch-icon.png'>

Esto garantiza que tus iconos estén controlados por versiones con tu código base y nunca dependan del estado de la base de datos.

Paso 3: Elimina el archivo adjunto del núcleo

Incluso con tus etiquetas Twig codificadas, Drupal aún podría inyectar su etiqueta de icono por defecto a través de los archivos adjuntos de la página, causando etiquetas duplicadas en tu DOM. Para eliminarlo, usamos un hook de preprocesamiento en tu archivo your_theme.theme.

/**
 * Implements hook_page_attachments_alter().
 */
function your_theme_page_attachments_alter(array &$attachments) {
  foreach ($attachments['#attached']['html_head_link'] as $key => $value) {
    if (isset($value[0]['rel']) && $value[0]['rel'] === 'icon') {
      unset($attachments['#attached']['html_head_link'][$key]);
    }
  }
}

Este fragmento intercepta el proceso de construcción de la página y elimina agresivamente cualquier enlace de icono que Drupal intente inyectar de forma nativa, dejando solo tu implementación limpia de Twig.

Borrando la caché de Drupal

Puedes hacer todos los cambios de código del mundo, pero Drupal no los mostrará hasta que borres la caché. La caché de renderizado es notoriamente obstinada.

  1. Abre tu terminal y navega a la raíz de tu proyecto.
  2. Ejecuta drush cr (Cache Rebuild).
  3. Si estás usando un proxy inverso como Varnish o un CDN como Cloudflare, purga esas cachés también.

Si aún ves la vieja gota azul, es probable que tu navegador la esté reteniendo. Lee nuestra guía sobre cómo forzar el borrado de la caché del favicon para resolver la obstinación del navegador local.

Errores comunes en Drupal 10/11

Si tu configuración sigue fallando, revisa estos culpables comunes:

  • Permisos de archivos: Asegúrate de que tu directorio /icons/ sea legible por el servidor web (generalmente 755 para directorios, 644 para archivos).
  • Problemas de ruta base: Si estás ejecutando Drupal en un subdirectorio, es posible que active_theme_path() necesite el prefijo de la URL base.
  • Herencia de temas: Si estás construyendo un subtema (por ejemplo, basado en Radix u Olivero), asegúrate de que tu tema principal no esté inyectando sus propios iconos a través de un hook de mayor prioridad.
