How to Add a Favicon for Vite App: The 2026 Dev Guide

If you've ever deployed a Vite app only to see a generic gray square in the browser tab, you know the frustration. It's not a bug; it's a configuration gap. Unlike older frameworks where you might have manually edited an HTML file, Vite relies on a strict file-based convention that often trips up developers migrating from Create React App.

When Stripe updated their branding, they didn't just swap one image file. They ensured every resolution, from the tiny 16x16 tab icon to the massive 512x512 PWA icon, was perfectly optimized. If you want your Vite app to look as professional as Stripe's dashboard, you need to stop guessing and start following the modern standard.

Why Vite Handles Icons Differently

Vite treats your public folder as a static asset root. This means files placed there are served exactly as they are, without processing. However, the magic happens when you name your files correctly. Vite automatically detects specific filenames like favicon.ico or favicon.svg and injects the necessary HTML tags during the build.

This approach eliminates the need for complex plugins or manual <link> tags in your index.html for the primary icon. It's a clean, opinionated workflow that saves time but requires precision. If you misspell the filename, the browser won't find the icon, and you'll be left with that dreaded default gray square.

Step 1: Prepare Your Source Files

Before touching your code, you need the right assets. A single PNG file is rarely enough for 2026 standards. You need an SVG for scalability and a multi-resolution ICO for legacy browser support. If you only have a logo, use Mzu favicondl to generate the necessary sizes instantly.

Ensure your SVG is optimized. Remove hidden layers and unnecessary metadata. A bloated SVG will slow down your initial paint, even if it's just a 16x16 pixel icon. For the ICO file, include 16x16, 32x32, and 48x48 sizes in a single container.

Step 2: Place Files in the Public Directory

Navigate to your project root. You will see a folder named public. This is the only place where static assets live. Drag and drop your favicon.svg and favicon.ico directly into this folder.

Do not put these files in src/assets. If you put them there, Vite will hash the filenames during the build process (e.g., favicon.a1b2c3.svg), which breaks the automatic detection mechanism. The public folder is your safe zone for files that must remain named exactly as they are.

Step 3: Verify the Configuration

Run your development server with npm run dev. Open your browser and inspect the page source. You should see a line similar to this:

<link rel="icon" type="image/svg+xml" href="/favicon.svg" />

If you see this, Vite has successfully detected your icon. If you don't, check your file extension. Vite is case-sensitive. Favicon.svg will not work; it must be favicon.svg. This is a common pitfall that wastes hours of debugging time.

Handling Dark Mode and Apple Touch Icons

A professional setup goes beyond the default tab icon. Modern users expect your site to adapt to their system preferences. If your brand uses a dark background, a white logo might vanish in dark mode. You need a specific SVG variant for this.

Implementing Dark Mode Support

Create a second SVG file named favicon-dark.svg. In your index.html (inside the public folder, not src), add a conditional link tag. This tells the browser to swap the icon based on the prefers-color-scheme media query.

<link rel="icon" type="image/svg+xml" href="/favicon.svg" media="(prefers-color-scheme: light)" /> <link rel="icon" type="image/svg+xml" href="/favicon-dark.svg" media="(prefers-color-scheme: dark)" />

This simple addition ensures your brand remains visible and polished regardless of the user's OS settings. It's a small detail that signals high-quality engineering.

Setting Up the Apple Touch Icon

Mobile users on iOS need a different icon. When they "Add to Home Screen," the browser expects a 180x180 PNG. Place a file named apple-touch-icon.png in your public folder. Vite will automatically serve this file when the browser requests it.

Do not forget the manifest file. Create a manifest.json in the public folder and link it in your index.html. This file tells Android and iOS how to display your app as a standalone PWA. For more details on the manifest structure, check our guide on Web App Manifest Favicon.

Common Pitfalls and Fixes

Even with the correct setup, things can go wrong. The most frequent issue is caching. Browsers are aggressive about caching favicons. If you update your icon but still see the old one, you aren't seeing a bug; you're seeing a cached version.

To force an update, you can add a query string to your link tag in index.html, like href="/favicon.svg?v=2". This tricks the browser into fetching the new file. Alternatively, clear your browser cache or use an incognito window to verify the change.

Another issue is the 404 error in the console. If your browser console shows a 404 for /favicon.ico, it means the file is missing from the public folder. Even if you use an SVG, some browsers still request the legacy ICO file. Always provide a fallback ICO file to silence these errors.

Finally, remember that Vite's build output goes to the dist folder. When you deploy, ensure your server serves the files from the public directory correctly. If you're using a static host like Vercel or Netlify, this happens automatically. If you're on a custom server, check your routing configuration.

Setting up a favicon for a Vite app doesn't need to be complicated. By respecting the file-based conventions and providing the right formats, you ensure your app looks professional across every device. It's the difference between a hobby project and a production-ready application.

Vite 处理图标的独特机制

Vite 将 public 目录视为静态资源根目录。放在这里的文件会原样输出，不会被处理。但关键在于文件名：只要命名正确，比如 favicon.ico 或 favicon.svg，Vite 就能在构建时自动检测并注入对应的 HTML 标签。

这种机制省去了安装复杂插件的麻烦，也无需在 index.html 里手动写 <link> 标签。这是一种“约定优于配置”的高效工作流，但容错率极低。文件名哪怕拼错一个字母，浏览器就找不到图标，你只能继续面对那个难看的灰色方块。

在动代码之前，先搞定资源。2026 年了，单靠一张 PNG 图是远远不够的。你需要一个 SVG 保证缩放不失真，还需要一个包含多分辨率的 ICO 文件以兼容老旧浏览器。如果你手头只有一个 Logo，可以用 Mzu favicondl 一键生成所需的各种尺寸。

深色模式与 Apple Touch Icon 的处理

专业的配置绝不止于默认标签页图标。现在的用户期望网站能适配系统偏好。如果你的品牌主色调是深色，在深色模式下，白色的 Logo 可能会直接“隐身”。这时候，你需要一个专门的 SVG 变体。

实现深色模式支持

创建第二个 SVG 文件，命名为 favicon-dark.svg。在你的 index.html 中（注意是 public 目录下的那个，不是 src 里的），添加条件链接标签。这能告诉浏览器根据 prefers-color-scheme 媒体查询自动切换图标。

这个简单的改动，能确保无论用户的系统设置如何，你的品牌标识始终清晰可见。这种细节，恰恰体现了工程的高质量。

配置 Apple Touch Icon

iOS 用户在“添加到主屏幕”时，浏览器需要一张 180x180 的 PNG 图标。请在 public 目录下放置一个名为 apple-touch-icon.png 的文件。Vite 会自动处理这个请求。

别忘了 manifest.json 文件。在 public 目录下创建它，并在 index.html 中引用。这个文件告诉 Android 和 iOS 如何将你的应用作为独立的 PWA 展示。关于 Manifest 结构的更多细节，可以参考我们的 Web App Manifest Favicon 指南。

常见陷阱与解决方案

即使配置正确，问题也可能出现。最常见的是缓存问题。浏览器对 favicon 的缓存非常激进。如果你更新了图标但看到的还是旧的，那不是 Bug，是缓存没清。

强制更新的一个小技巧是在 index.html 的链接标签后加查询参数，比如 href="/favicon.svg?v=2"。这能骗过浏览器去请求新文件。或者，直接清除浏览器缓存，或用无痕模式验证。

另一个常见问题是控制台报 404。如果浏览器控制台显示 /favicon.ico 404，说明 public 目录下缺少该文件。即使你主要用 SVG，部分浏览器依然会请求传统的 ICO 文件。务必提供一个 ICO 作为兜底，以消除这些报错。

最后，记住 Vite 的构建产物在 dist 目录。部署时，确保服务器能正确提供 public 目录下的文件。如果你用的是 Vercel 或 Netlify 这类静态托管服务，这通常是自动完成的。如果是自定义服务器，请检查路由配置。

Vite がアイコンを扱う仕組みの違い

Vite は public フォルダを静的アセットのルートとして扱います。つまり、ここに置かれたファイルは処理されず、そのまま提供されます。しかし、魔法が起きるのはファイル名を正しくつけたときです。Vite は favicon.ico や favicon.svg といった特定のファイル名を自動検出し、ビルド時に必要な HTML タグを注入します。

このアプローチにより、メインのアイコンに対して複雑なプラグインを使ったり、index.html に手動で <link> タグを追加したりする必要がなくなります。これは時間を節約するクリーンで意見のあるワークフローですが、正確さが求められます。ファイル名のスペルを間違えると、ブラウザはアイコンを見つけられず、あの忌まわしいデフォルトのグレーの四角が表示されてしまいます。

ステップ 1: ソースファイルの準備

コードに触れる前に、適切なアセットを用意しましょう。2026 年の基準では、単一の PNG ファイルだけでは不十分です。スケーラビリティのための SVG と、レガシーブラウザ対応のためのマルチ解像度 ICO が必要です。ロゴしかない場合は、Mzu favicondl を使って必要なサイズを瞬時に生成しましょう。

SVG は最適化されていることを確認してください。隠れたレイヤーや不要なメタデータは削除しましょう。たとえ 16x16 ピクセルのアイコンであっても、肥大化した SVG は初期描画を遅くします。ICO ファイルには、16x16、32x32、48x48 のサイズを単一のコンテナに含めてください。

ステップ 2: ファイルを Public ディレクトリに配置

プロジェクトのルートディレクトリに移動します。public という名前のフォルダが表示されるはずです。これが静的アセットが存在する唯一の場所です。favicon.svg と favicon.ico をこのフォルダに直接ドラッグ＆ドロップしてください。

これらのファイルを src/assets に置いてはいけません。そこに置くと、ビルド処理中に Vite がファイル名にハッシュを付与します（例：favicon.a1b2c3.svg）。これにより自動検出メカニズムが機能しなくなります。ファイル名を正確に保つ必要がある場合は、public フォルダが安全地帯です。

ステップ 3: 設定の確認

npm run dev で開発サーバーを実行します。ブラウザを開いてページのソースを確認してください。以下のような行が表示されているはずです：

<link rel="icon" type="image/svg+xml" href="/favicon.svg" />

これが表示されていれば、Vite はアイコンを正常に検出しています。表示されない場合は、ファイル拡張子を確認してください。Vite は大文字小文字を区別します。Favicon.svg は機能せず、favicon.svg でなければなりません。これは数時間のデバッグ時間を無駄にする一般的な落とし穴です。

ダークモードと Apple Touch Icon への対応

プロフェッショナルな設定は、デフォルトのタブアイコン以上のものです。現代のユーザーは、システム設定に合わせてサイトが適応することを期待しています。ブランドがダーク背景を使用している場合、ダークモードでは白いロゴが見えなくなることがあります。これには専用の SVG バリエーションが必要です。

ダークモード対応の実装

favicon-dark.svg という名前の 2 つ目の SVG ファイルを作成します。index.html（src ではなく public フォルダ内）に、条件付きのリンクタグを追加します。これにより、ブラウザは prefers-color-scheme メディアクエリに基づいてアイコンを切り替えます。

<link rel="icon" type="image/svg+xml" href="/favicon.svg" media="(prefers-color-scheme: light)" /> <link rel="icon" type="image/svg+xml" href="/favicon-dark.svg" media="(prefers-color-scheme: dark)" />

このシンプルな追加により、ユーザーの OS 設定に関係なく、ブランドが鮮明で洗練された状態で表示されます。これは高品質なエンジニアリングをアピールする小さな詳細です。

Apple Touch Icon の設定

iOS のモバイルユーザーには異なるアイコンが必要です。「ホーム画面に追加」された際、ブラウザは 180x180 の PNG を期待します。public フォルダに apple-touch-icon.png という名前のファイルを配置してください。ブラウザがリクエストすると、Vite は自動的にこのファイルを提供します。

マニフェストファイルも忘れずに。public フォルダに manifest.json を作成し、index.html でリンクしてください。このファイルは、Android や iOS にアプリをスタンドアロンの PWA として表示する方法を伝えます。マニフェスト構造の詳細については、Web App Manifest Favicon のガイドをご覧ください。

一般的な落とし穴と解決策

設定が正しくても、問題は起こり得ます。最も頻繁な問題はキャッシュです。ブラウザは favicon のキャッシュに対して非常に積極的です。アイコンを更新しても古いまま表示される場合、バグではなくキャッシュされたバージョンを見ています。

更新を強制するには、index.html のリンクタグにクエリ文字列を追加します。例：href="/favicon.svg?v=2"。これでブラウザに新しいファイルを取得させることができます。あるいは、ブラウザのキャッシュをクリアするか、シークレットウィンドウを使用して変更を確認してください。

もう一つの問題は、コンソールに表示される 404 エラーです。ブラウザコンソールに /favicon.ico の 404 エラーが表示される場合、public フォルダからファイルが欠落していることを意味します。SVG を使用していても、一部のブラウザはレガシーな ICO ファイルをリクエストします。これらのエラーを沈黙させるために、必ずフォールバック用の ICO ファイルを提供してください。

最後に、Vite のビルド出力は dist フォルダに出力されることを忘れないでください。デプロイする際は、サーバーが public ディレクトリからファイルを正しく提供していることを確認してください。Vercel や Netlify などの静的ホストを使用している場合、これは自動的に処理されます。カスタムサーバーを使用している場合は、ルーティング設定を確認してください。

Vite アプリの favicon 設定は複雑である必要はありません。ファイルベースの規約を尊重し、適切な形式を提供することで、すべてのデバイスでプロフェッショナルな外観を保証できます。これは趣味のプロジェクトと、本番環境で使えるアプリケーションの違いです。

Vite 가 아이콘을 다르게 처리하는 이유

Vite 는 public 폴더를 정적 자산 루트로 간주합니다. 이 폴더에 배치된 파일은 처리 없이 그대로 제공됩니다. 하지만 마법은 파일 이름을 올바르게 지정했을 때 발생합니다. Vite 는 favicon.ico 또는 favicon.svg 같은 특정 파일명을 자동으로 감지하여 빌드 시 필요한 HTML 태그를 주입합니다.

이 방식은 복잡한 플러그인이나 index.html에 수동으로 <link> 태그를 추가할 필요를 없앱니다. 깔끔하고 의견이 반영된 워크플로우지만, 정확성이 요구됩니다. 파일 이름을 철자 실수만 해도 브라우저가 아이콘을 찾지 못해 dreaded 회색 사각형이 남게 됩니다.

1 단계: 소스 파일 준비

코드를 건드리기 전에 올바른 자산을 준비해야 합니다. 2026 년 기준 단일 PNG 파일로는 부족합니다. 확장성을 위한 SVG 와 레거시 브라우저 지원을 위한 다중 해상도 ICO 가 필요합니다. 로고만 있다면 Mzu favicondl을 사용해 즉시 필요한 크기를 생성하세요.

SVG 는 최적화되어 있어야 합니다. 숨겨진 레이어와 불필요한 메타데이터를 제거하세요. 16x16 픽셀 아이콘이라 하더라도 무거운 SVG 는 초기 렌더링을 늦출 수 있습니다. ICO 파일에는 16x16, 32x32, 48x48 크기를 단일 컨테이너에 포함시키세요.

2 단계: Public 디렉토리에 파일 배치

프로젝트 루트로 이동하세요. public이라는 폴더가 보일 겁니다. 정적 자산이 존재하는 유일한 곳입니다. favicon.svg와 favicon.ico를 이 폴더로 직접 드래그 앤 드롭하세요.

이 파일들을 src/assets에 넣지 마세요. 그곳에 넣으면 빌드 과정에서 Vite 가 파일명에 해시값을 추가합니다 (예: favicon.a1b2c3.svg). 이는 자동 감지 메커니즘을 무효화합니다. public 폴더는 파일명이 그대로 유지되어야 하는 파일들의 안전지대입니다.

3 단계: 설정 확인

npm run dev로 개발 서버를 실행하세요. 브라우저에서 페이지 소스를 확인하세요. 다음과 유사한 라인이 표시되어야 합니다:

<link rel="icon" type="image/svg+xml" href="/favicon.svg" />

이것이 보인다면 Vite 가 아이콘을 성공적으로 감지한 것입니다. 보이지 않는다면 파일 확장자를 확인하세요. Vite 는 대소문자를 구분합니다. Favicon.svg는 작동하지 않으며 반드시 favicon.svg여야 합니다. 이는 수 시간의 디버깅 시간을 낭비하게 하는 흔한 함정입니다.

다크 모드 및 Apple Touch Icon 처리

프로페셔널한 설정은 기본 탭 아이콘을 넘어섭니다. 현대 사용자는 사이트가 시스템 환경설정에 적응하기를 기대합니다. 브랜드가 다크 배경을 사용한다면, 다크 모드에서 흰색 로고가 사라질 수 있습니다. 이를 위한 전용 SVG 변형이 필요합니다.

다크 모드 지원 구현

favicon-dark.svg라는 두 번째 SVG 파일을 생성하세요. index.html(src가 아닌 public 폴더 내부) 에 조건부 링크 태그를 추가하세요. 이는 브라우저가 prefers-color-scheme 미디어 쿼리에 따라 아이콘을 교체하도록 지시합니다.

<link rel="icon" type="image/svg+xml" href="/favicon.svg" media="(prefers-color-scheme: light)" /> <link rel="icon" type="image/svg+xml" href="/favicon-dark.svg" media="(prefers-color-scheme: dark)" />

이 간단한 추가는 사용자의 OS 설정과 관계없이 브랜드가 선명하고 세련되게 유지되도록 보장합니다. 이는 고품질 엔지니어링을 알리는 작은 디테일입니다.

Apple Touch Icon 설정

iOS 모바일 사용자는 다른 아이콘이 필요합니다. 홈 화면에 추가할 때 브라우저는 180x180 PNG 를 기대합니다. public 폴더에 apple-touch-icon.png라는 파일을 배치하세요. Vite 는 브라우저가 요청할 때 이 파일을 자동으로 제공합니다.

매니페스트 파일을 잊지 마세요. public 폴더에 manifest.json을 생성하고 index.html에 링크하세요. 이 파일은 Android 와 iOS 에게 앱을 독립적인 PWA 로 표시하는 방법을 알려줍니다. 매니페스트 구조에 대한 자세한 내용은 Web App Manifest Favicon 가이드를 확인하세요.

흔한 함정 및 해결책

설정이 정확하더라도 문제가 발생할 수 있습니다. 가장 빈번한 문제는 캐싱입니다. 브라우저는 favicon 캐싱에 매우 공격적입니다. 아이콘을 업데이트했는데도 이전 것이 보인다면 버그가 아니라 캐시된 버전을 보고 있는 것입니다.

업데이트를 강제하려면 index.html의 링크 태그에 쿼리 문자열을 추가하세요. 예를 들어 href="/favicon.svg?v=2"처럼요. 이는 브라우저가 새 파일을 가져오도록 속입니다. 또는 브라우저 캐시를 지우거나 시크릿 창을 사용하여 변경 사항을 확인하세요.

또 다른 문제는 콘솔의 404 오류입니다. 브라우저 콘솔에 /favicon.ico에 대한 404 오류가 표시된다면 public 폴더에서 파일이 누락된 것입니다. SVG 를 사용하더라도 일부 브라우저는 여전히 레거시 ICO 파일을 요청합니다. 이러한 오류를 침묵시키기 위해 항상 대체 ICO 파일을 제공하세요.

마지막으로, Vite 빌드 출력물이 dist 폴더로 간다는 점을 기억하세요. 배포할 때 서버가 public 디렉토리의 파일을 올바르게 제공하는지 확인하세요. Vercel 이나 Netlify 같은 정적 호스팅을 사용한다면 자동으로 처리됩니다. 커스텀 서버를 사용한다면 라우팅 설정을 점검하세요.

Vite 앱에 favicon 을 설정하는 것이 복잡할 필요는 없습니다. 파일 기반 컨벤션을 존중하고 올바른 포맷을 제공하면 모든 기기에서 앱이 프로페셔널하게 보이도록 보장할 수 있습니다. 이는 취미 프로젝트와 프로덕션 준비 완료 애플리케이션의 차이입니다.

Si alguna vez has desplegado una app de Vite y te has encontrado con ese cuadrado gris genérico en la pestaña del navegador, sabes de lo que hablo. No es un bug, es un hueco de configuración. A diferencia de frameworks más antiguos donde podías editar manualmente el HTML, Vite se basa en una convención estricta de nombres de archivo que suele pillar desprevenidos a los que vienen de Create React App.

Cuando Mercadona actualizó su branding, no se limitó a cambiar una imagen. Se aseguraron de que cada resolución, desde el icono de 16x16 de la pestaña hasta el de 512x512 para la PWA, estuviera perfectamente optimizada. Si quieres que tu app de Vite tenga un aspecto tan profesional como el panel de control de Mercadona, tienes que dejar de hacer pruebas y seguir el estándar moderno.

Por qué Vite gestiona los iconos de forma diferente

Vite trata tu carpeta public como la raíz de los assets estáticos. Esto significa que los archivos colocados ahí se sirven tal cual, sin procesar. Sin embargo, la magia ocurre cuando nombras tus archivos correctamente. Vite detecta automáticamente nombres de archivo específicos como favicon.ico o favicon.svg e inyecta las etiquetas HTML necesarias durante el build.

Este enfoque elimina la necesidad de plugins complejos o etiquetas <link> manuales en tu index.html para el icono principal. Es un flujo de trabajo limpio y con opinión que ahorra tiempo, pero requiere precisión. Si escribes mal el nombre del archivo, el navegador no encontrará el icono y te quedarás con ese temido cuadrado gris por defecto.

Paso 1: Prepara tus archivos fuente

Antes de tocar el código, necesitas los assets adecuados. Un solo archivo PNG rara vez es suficiente para los estándares de 2026. Necesitas un SVG para la escalabilidad y un ICO multiresolución para navegadores antiguos. Si solo tienes un logo, usa Mzu favicondl para generar los tamaños necesarios al instante.

Asegúrate de que tu SVG esté optimizado. Elimina capas ocultas y metadatos innecesarios. Un SVG hinchado ralentizará tu pintado inicial, aunque sea solo un icono de 16x16 píxeles. Para el archivo ICO, incluye los tamaños 16x16, 32x32 y 48x48 en un solo contenedor.

Paso 2: Coloca los archivos en el directorio public

Navega hasta la raíz de tu proyecto. Verás una carpeta llamada public. Este es el único lugar donde viven los assets estáticos. Arrastra y suelta tus archivos favicon.svg y favicon.ico directamente en esta carpeta.

No pongas estos archivos en src/assets. Si los pones ahí, Vite hasheará los nombres de archivo durante el proceso de build (por ejemplo, favicon.a1b2c3.svg), lo que rompe el mecanismo de detección automática. La carpeta public es tu zona segura para archivos que deben permanecer con el nombre exacto que tienen.

Paso 3: Verifica la configuración

Ejecuta tu servidor de desarrollo con npm run dev. Abre tu navegador e inspecciona el código fuente de la página. Deberías ver una línea similar a esta:

<link rel='icon' type='image/svg+xml' href='/favicon.svg' />

Si ves esto, Vite ha detectado tu icono correctamente. Si no, comprueba la extensión de tu archivo. Vite distingue entre mayúsculas y minúsculas. Favicon.svg no funcionará; debe ser favicon.svg. Este es un error común que hace perder horas de depuración.

Cómo manejar el modo oscuro y los iconos de Apple Touch

Una configuración profesional va más allá del icono de pestaña por defecto. Los usuarios modernos esperan que tu sitio se adapte a sus preferencias del sistema. Si tu marca usa un fondo oscuro, un logo blanco podría desaparecer en modo oscuro. Necesitas una variante SVG específica para esto.

Implementando soporte para modo oscuro

Crea un segundo archivo SVG llamado favicon-dark.svg. En tu index.html (dentro de la carpeta public, no en src), añade una etiqueta link condicional. Esto le dice al navegador que intercambie el icono según la media query prefers-color-scheme.

<link rel='icon' type='image/svg+xml' href='/favicon.svg' media='(prefers-color-scheme: light)' /> <link rel='icon' type='image/svg+xml' href='/favicon-dark.svg' media='(prefers-color-scheme: dark)' />

Esta simple adición asegura que tu marca permanezca visible y pulida independientemente de la configuración del sistema operativo del usuario. Es un pequeño detalle que denota ingeniería de alta calidad.

Configurando el icono de Apple Touch

Los usuarios móviles en iOS necesitan un icono diferente. Cuando añaden tu sitio a la pantalla de inicio, el navegador espera un PNG de 180x180. Coloca un archivo llamado apple-touch-icon.png en tu carpeta public. Vite servirá este archivo automáticamente cuando el navegador lo solicite.

No olvides el archivo de manifiesto. Crea un manifest.json en la carpeta public y enlázalo en tu index.html. Este archivo le dice a Android e iOS cómo mostrar tu app como una PWA independiente. Para más detalles sobre la estructura del manifiesto, consulta nuestra guía sobre Web App Manifest Favicon.

Problemas comunes y soluciones

Incluso con la configuración correcta, las cosas pueden salir mal. El problema más frecuente es la caché. Los navegadores son agresivos al cachear los favicons. Si actualizas tu icono pero sigues viendo el antiguo, no estás viendo un bug; estás viendo una versión en caché.

Para forzar una actualización, puedes añadir una cadena de consulta a tu etiqueta link en index.html, como href='/favicon.svg?v=2'. Esto engaña al navegador para que obtenga el nuevo archivo. Alternativamente, limpia la caché de tu navegador o usa una ventana de incógnito para verificar el cambio.

Otro problema es el error 404 en la consola. Si la consola de tu navegador muestra un 404 para /favicon.ico, significa que el archivo falta en la carpeta public. Incluso si usas un SVG, algunos navegadores siguen solicitando el archivo ICO heredado. Proporciona siempre un archivo ICO de respaldo para silenciar estos errores.

Finalmente, recuerda que la salida del build de Vite va a la carpeta dist. Cuando despliegues, asegúrate de que tu servidor sirva los archivos del directorio public correctamente. Si usas un host estático como Vercel o Netlify, esto ocurre automáticamente. Si estás en un servidor personalizado, revisa tu configuración de enrutamiento.

Es la diferencia entre un proyecto de hobby y una aplicación lista para producción.
