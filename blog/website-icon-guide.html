Website Icon Complete Guide: The 2026 Developer Stack

If you've ever deployed a beautiful web application, proudly saved it to your iPhone home screen, and stared in horror at a blurry, auto-generated letter instead of your logo—you need a website icon complete guide to fix your setup.

We have all been there. You finish building a site, realize you forgot the favicon, and hastily drop a 16x16 pixel favicon.ico into your root directory. Ten years ago, that was enough. In 2026, doing that guarantees your brand will look broken on at least half the devices your users own.

The browser ecosystem has fractured. Your icon doesn't just live in a 16-pixel browser tab anymore. It lives on iOS home screens, Android app drawers, Windows start menus, Safari pinned tabs, and Google search results. If you rely on legacy methods, you are actively harming your site's professional appearance and click-through rates.

The Website Icon Complete Guide: Your 2026 Stack

Stop generating 30 different PNG sizes. It is a massive waste of bandwidth and developer time. Modern browsers have coalesced around a much simpler, more powerful standard. You only need four specific files to cover 99% of all devices and browsers.

1. The Modern Foundation (SVG)

The core of your icon strategy should now be an SVG favicon. Scalable Vector Graphics are resolution-independent, meaning a single file looks razor-sharp on a low-end laptop and a 6K Retina display. More importantly, SVG allows for embedded CSS.

Why does embedded CSS matter? Dark mode. You can write a media query directly inside your SVG file so your icon changes color based on the user's system theme. If you have a black logo, it will automatically turn white when a user switches their browser to dark mode. No JavaScript required.

2. The Apple Ecosystem (Apple Touch Icon)

Apple devices notoriously ignore standard web manifests for home screen icons. If a user taps 'Add to Home Screen' on Safari, iOS looks for a very specific file: the Apple Touch Icon.

You need a single 180x180 pixel PNG. Do not use a transparent background for this file. iOS will automatically fill any transparency with black, which usually ruins your logo. Give it a solid background color and ensure your logo is centered with plenty of padding.

3. The Android & PWA World (Web App Manifest)

Android devices and Progressive Web Apps (PWAs) rely on a site.webmanifest file. This is a simple JSON file that tells the operating system how your web app should behave when installed.

Inside this manifest, you define your icons. You need exactly two PNG sizes here: 192x192 and 512x512. The 192px version is used for home screen icons, while the 512px version is used for splash screens when the app loads. You also need to declare these as 'maskable' so Android can shape them into circles, teardrops, or rounded squares depending on the user's launcher preferences.

4. The Legacy Fallback (ICO)

We still need a favicon.ico file. Why? Because legacy browsers, RSS readers, and some enterprise web scrapers still hardcode requests to /favicon.ico. If you don't have it, your server logs will fill up with annoying 404 errors.

Keep this file small. A single 32x32 pixel ICO is perfect. It provides enough detail for legacy systems without bloating your initial page load.

Real-World Example: How GitHub Nails It

Look at GitHub. If you open GitHub in a browser tab and toggle your operating system between light and dark mode, the little Octocat icon instantly flips from black to white. They don't use complex JavaScript to swap the DOM elements.

They use a single SVG favicon with a prefers-color-scheme media query. Here is a simplified version of exactly what they do inside their SVG:

<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 100 100'>
  <style>
    path { fill: #24292f; }
    @media (prefers-color-scheme: dark) {
      path { fill: #ffffff; }
    }
  </style>
  <path d='...' />
</svg>

This is elegant, lightweight, and entirely handled by the browser's rendering engine. It is the gold standard for 2026 web development.

The Code: Tying It All Together

You have your four files: icon.svg, apple-touch-icon.png, icon-192.png, and icon-512.png (referenced in your manifest). How do you actually implement them in your HTML?

Add this exact snippet to the <head> of your document. Order matters here. Browsers read from top to bottom and will generally use the last compatible icon they find.

<!-- 1. The legacy fallback (keep it in the root directory) -->
<link rel='icon' href='/favicon.ico' sizes='32x32'>

<!-- 2. The modern SVG -->
<link rel='icon' href='/icon.svg' type='image/svg+xml'>

<!-- 3. The Apple Touch Icon -->
<link rel='apple-touch-icon' href='/apple-touch-icon.png'>

<!-- 4. The Web Manifest -->
<link rel='manifest' href='/site.webmanifest'>

Notice what is missing? There are no 16x16 PNGs. There are no Windows Metro tile meta tags. Microsoft Edge now uses the Web App Manifest, making those old proprietary tags obsolete. We have finally reached a point where less code equals better coverage.

Best Practices You Are Probably Ignoring

Even with the right files, developers frequently make design and implementation mistakes. Here are the hard-won lessons from auditing thousands of sites.

Mind the Maskable Safe Zone

When you define your 192x192 and 512x512 icons in your manifest, you should include "purpose": "any maskable". However, Android will aggressively crop your icon to fit different shapes. If your logo stretches to the absolute edge of the PNG, the corners will be cut off.

  • Keep all critical visual elements within the inner 80% of the image.
  • Use a solid background color. Transparent maskable icons will render with a harsh white or black background depending on the device theme.
  • Test your icon using Chrome DevTools under the Application > Manifest tab to see how it looks cropped.

Cache Busting is Mandatory

Favicons are the most aggressively cached assets on the internet. Browsers hate downloading them twice. If you update your logo and just replace the file on your server, your regular users might not see the new icon for months.

Always use cache-busting query strings or file hashes when you update your icons. Change href='/icon.svg' to href='/icon.svg?v=2'. This forces the browser to fetch the fresh asset immediately.

Optimize for Google Search

Google displays your favicon next to your site's title in mobile and desktop search results. A high-contrast, recognizable icon significantly boosts your Click-Through Rate (CTR). Google specifically requires your icon to be a multiple of 48px square (e.g., 48x48, 96x96, 144x144) or an SVG. Our setup above satisfies this requirement perfectly through the scalable SVG.

别再生成30种不同尺寸的PNG了。那是对带宽和开发者时间的巨大浪费。现代浏览器已经统一到了一套更简单、更强大的标准上。你只需要四个特定的文件，就能覆盖99%的设备和浏览器。

你图标策略的核心现在应该是SVG favicon。可缩放矢量图形与分辨率无关，这意味着同一个文件在低端笔记本和6K视网膜屏幕上都能保持锐利。更重要的是，SVG允许内嵌CSS。

内嵌CSS为什么重要？为了暗黑模式。你可以直接在SVG文件中编写媒体查询，让图标根据用户的系统主题改变颜色。如果你的Logo是黑色的，当用户切换到暗黑模式时，它可以自动变成白色。完全不需要JavaScript。

众所周知，苹果设备在处理主屏幕图标时会忽略标准的Web Manifest。如果用户在Safari中点击“添加到主屏幕”，iOS会寻找一个非常特定的文件：Apple Touch Icon。

3. Android与PWA世界 (Web App Manifest)

Android设备和渐进式Web应用（PWA）依赖于site.webmanifest文件。这是一个简单的JSON文件，告诉操作系统你的Web应用在安装后应该如何表现。

在这个清单中，你需要定义你的图标。这里只需要两个PNG尺寸：192x192和512x512。192px版本用于主屏幕图标，而512px版本用于应用加载时的启动画面。你还需要将它们声明为'maskable'（可遮罩），这样Android就可以根据用户的桌面启动器偏好，将它们裁剪成圆形、水滴形或圆角矩形。

4. 传统后备方案 (ICO)

我们仍然需要一个favicon.ico文件。为什么？因为老旧浏览器、RSS阅读器以及一些企业级网络爬虫仍然会硬编码请求/favicon.ico。如果没有它，你的服务器日志将被烦人的404错误填满。

保持这个文件尽可能小。一个32x32像素的ICO就足够了。它为传统系统提供了足够的细节，又不会拖慢初始页面加载速度。

  • 将所有关键视觉元素保持在图像内部的80%区域内。

  • 使用Chrome开发者工具的Application > Manifest选项卡测试你的图标，查看裁剪后的效果。

ウェブサイトアイコン完全ガイド：2026年のスタック

30種類もの異なるPNGサイズを生成するのはやめましょう。帯域幅と開発者の時間の無駄です。最新のブラウザは、よりシンプルで強力な標準に統合されています。99%のデバイスとブラウザをカバーするには、4つの特定のファイルだけで十分です。

1. モダンな基盤 (SVG)

現在のアイコン戦略の中核はSVGファビコンであるべきです。スケーラブル・ベクター・グラフィックスは解像度に依存しないため、単一のファイルで低スペックのノートPCでも6KのRetinaディスプレイでも非常にシャープに見えます。さらに重要なのは、SVGにはCSSを埋め込めることです。

なぜ埋め込みCSSが重要なのでしょうか？それはダークモードのためです。SVGファイル内に直接メディアクエリを記述できるため、ユーザーのシステムテーマに基づいてアイコンの色を変更できます。黒いロゴがある場合、ユーザーがブラウザをダークモードに切り替えると自動的に白になります。JavaScriptは一切不要です。

2. Appleエコシステム (Apple Touch Icon)

Appleデバイスは、ホーム画面のアイコンに関して標準のWebマニフェストを無視することで有名です。ユーザーがSafariで「ホーム画面に追加」をタップすると、iOSは非常に特定のファイルを探します。それがApple Touch Iconです。

180x180ピクセルのPNGが1つ必要です。このファイルには透明な背景を使用しないでください。iOSは透明部分を自動的に黒で塗りつぶすため、通常はロゴが台無しになります。単色の背景色を指定し、ロゴが中央に配置され、十分な余白があることを確認してください。

3. AndroidとPWAの世界 (Web App Manifest)

Androidデバイスとプログレッシブウェブアプリ（PWA）は、site.webmanifestファイルに依存しています。これは、インストール時にウェブアプリがどのように動作すべきかをオペレーティングシステムに伝えるシンプルなJSONファイルです。

このマニフェスト内でアイコンを定義します。ここでは、192x192と512x512の正確に2つのPNGサイズが必要です。192pxバージョンはホーム画面のアイコンに使用され、512pxバージョンはアプリの読み込み時のスプラッシュ画面に使用されます。また、Androidがユーザーのランチャーの好みに応じて円、しずく型、角丸四角形に形作れるように、これらを「maskable（マスク可能）」として宣言する必要があります。

4. レガシーフォールバック (ICO)

それでもfavicon.icoファイルは必要です。なぜでしょうか？古いブラウザ、RSSリーダー、および一部のエンタープライズウェブスクレイパーは、依然として/favicon.icoへのリクエストをハードコードしているからです。これがないと、サーバーのログが厄介な404エラーで埋め尽くされます。

このファイルは小さく保ちましょう。32x32ピクセルのICOが1つあれば完璧です。初期ページの読み込みを遅くすることなく、レガシーシステムに十分な詳細を提供します。

マスク可能なセーフゾーンに注意

マニフェストで192x192と512x512のアイコンを定義する際、"purpose": "any maskable"を含める必要があります。しかし、Androidは異なる形状に合わせるためにアイコンを積極的にトリミングします。ロゴがPNGの絶対的な端まで伸びていると、角が切り落とされてしまいます。

  • 重要な視覚要素はすべて、画像の内側80%に収めてください。
  • 単色の背景色を使用してください。透明なマスク可能アイコンは、デバイスのテーマに応じて厳しい白または黒の背景でレンダリングされます。
  • Chrome DevToolsの「Application」>「Manifest」タブを使用してアイコンをテストし、トリミングされたときにどのように見えるかを確認してください。

キャッシュバスティングは必須

ファビコンはインターネット上で最も積極的にキャッシュされるアセットです。ブラウザはそれらを2回ダウンロードするのを嫌います。ロゴを更新してサーバー上のファイルを単に置き換えただけでは、通常のユーザーは何ヶ月も新しいアイコンを見られない可能性があります。

アイコンを更新する際は、必ずキャッシュバスティングのクエリ文字列やファイルハッシュを使用してください。href='/icon.svg'をhref='/icon.svg?v=2'に変更します。これにより、ブラウザは直ちに新しいアセットを取得するよう強制されます。

브라우저 생태계는 파편화되었습니다. 아이콘은 더 이상 16픽셀 브라우저 탭에만 존재하지 않습니다. iOS 홈 화면, Android 앱 서랍, Windows 시작 메뉴, Safari 고정 탭, 그리고 Google 검색 결과에도 나타납니다. 과거의 방식에 의존한다면 사이트의 전문적인 외관과 클릭률을 적극적으로 훼손하는 셈입니다.

웹사이트 아이콘 완벽 가이드: 2026년 스택

30가지나 되는 다양한 크기의 PNG를 생성하는 일은 이제 그만두세요. 대역폭과 개발자 시간의 엄청난 낭비입니다. 최신 브라우저들은 훨씬 더 간단하고 강력한 표준으로 통합되었습니다. 99%의 기기와 브라우저를 커버하려면 단 4개의 특정 파일만 있으면 됩니다.

1. 모던 파운데이션 (SVG)

이제 아이콘 전략의 핵심은 SVG 파비콘이어야 합니다. 확장 가능한 벡터 그래픽(SVG)은 해상도에 독립적이므로, 단일 파일로 저사양 노트북부터 6K Retina 디스플레이까지 선명하게 보입니다. 더 중요한 점은 SVG가 내장 CSS를 허용한다는 것입니다.

내장 CSS가 왜 중요할까요? 바로 다크 모드 때문입니다. SVG 파일 내부에 직접 미디어 쿼리를 작성하여 사용자의 시스템 테마에 따라 아이콘 색상이 변경되도록 할 수 있습니다. 검은색 로고가 있다면, 사용자가 브라우저를 다크 모드로 전환할 때 자동으로 흰색으로 바뀝니다. JavaScript가 전혀 필요하지 않습니다.

2. Apple 생태계 (Apple Touch Icon)

Apple 기기들은 홈 화면 아이콘에 대해 표준 웹 매니페스트를 무시하는 것으로 악명이 높습니다. 사용자가 Safari에서 '홈 화면에 추가'를 탭하면, iOS는 매우 구체적인 파일을 찾습니다. 바로 Apple Touch Icon입니다.

180x180 픽셀의 단일 PNG가 필요합니다. 이 파일에는 투명한 배경을 사용하지 마세요. iOS는 투명한 부분을 자동으로 검은색으로 채우며, 이는 보통 로고를 망치게 됩니다. 단색 배경을 지정하고 로고가 충분한 여백과 함께 중앙에 오도록 하세요.

3. Android 및 PWA 세계 (Web App Manifest)

Android 기기와 프로그레시브 웹 앱(PWA)은 site.webmanifest 파일에 의존합니다. 이는 웹 앱이 설치되었을 때 운영 체제에 어떻게 작동해야 하는지 알려주는 간단한 JSON 파일입니다.

이 매니페스트 내에서 아이콘을 정의합니다. 여기에는 정확히 192x192와 512x512 두 가지 PNG 크기가 필요합니다. 192px 버전은 홈 화면 아이콘에 사용되고, 512px 버전은 앱 로드 시 스플래시 화면에 사용됩니다. 또한 Android가 사용자의 런처 설정에 따라 원, 물방울, 둥근 사각형 모양으로 자를 수 있도록 이를 'maskable(마스킹 가능)'로 선언해야 합니다.

4. 레거시 폴백 (ICO)

여전히 favicon.ico 파일은 필요합니다. 왜일까요? 구형 브라우저, RSS 리더, 일부 엔터프라이즈 웹 스크래퍼가 여전히 /favicon.ico에 대한 요청을 하드코딩하고 있기 때문입니다. 이 파일이 없으면 서버 로그가 성가신 404 에러로 가득 찰 것입니다.

이 파일은 작게 유지하세요. 32x32 픽셀의 단일 ICO면 완벽합니다. 초기 페이지 로드 속도를 늦추지 않으면서 레거시 시스템에 충분한 디테일을 제공합니다.

실제 사례: GitHub의 완벽한 적용

GitHub를 살펴보세요. 브라우저 탭에서 GitHub를 열고 운영 체제를 라이트 모드와 다크 모드 사이에서 전환하면, 작은 Octocat 아이콘이 즉시 검은색에서 흰색으로 바뀝니다. 그들은 DOM 요소를 교체하기 위해 복잡한 JavaScript를 사용하지 않습니다.

그들은 prefers-color-scheme 미디어 쿼리가 포함된 단일 SVG 파비콘을 사용합니다. 다음은 그들이 SVG 내부에서 정확히 수행하는 작업의 단순화된 버전입니다.

<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 100 100'>
  <style>
    path { fill: #24292f; }
    @media (prefers-color-scheme: dark) {
      path { fill: #ffffff; }
    }
  </style>
  <path d='...' />
</svg>

이 방식은 우아하고 가벼우며 브라우저의 렌더링 엔진에 의해 전적으로 처리됩니다. 이것이 2026년 웹 개발의 골드 스탠다드입니다.

코드: 모두 하나로 묶기

이제 icon.svg, apple-touch-icon.png, icon-192.png, icon-512.png(매니페스트에서 참조)라는 네 개의 파일이 있습니다. 이를 HTML에 어떻게 구현할까요?

문서의 <head>에 이 정확한 스니펫을 추가하세요. 순서가 중요합니다. 브라우저는 위에서 아래로 읽으며 일반적으로 찾은 마지막 호환 아이콘을 사용합니다.

<!-- 1. 레거시 폴백 (루트 디렉토리에 유지) -->
<link rel='icon' href='/favicon.ico' sizes='32x32'>

<!-- 2. 모던 SVG -->
<link rel='icon' href='/icon.svg' type='image/svg+xml'>

<!-- 3. Apple Touch Icon -->
<link rel='apple-touch-icon' href='/apple-touch-icon.png'>

<!-- 4. 웹 매니페스트 -->
<link rel='manifest' href='/site.webmanifest'>

무엇이 빠졌는지 눈치채셨나요? 16x16 PNG가 없습니다. Windows Metro 타일 메타 태그도 없습니다. Microsoft Edge는 이제 Web App Manifest를 사용하므로 이전의 독점 태그는 더 이상 필요하지 않습니다. 마침내 코드 양이 적을수록 더 나은 커버리지를 제공하는 단계에 도달했습니다.

당신이 놓치고 있을지 모르는 모범 사례

마스킹 가능한 안전 영역 주의

매니페스트에서 192x192 및 512x512 아이콘을 정의할 때 "purpose": "any maskable"을 포함해야 합니다. 그러나 Android는 다양한 모양에 맞추기 위해 아이콘을 공격적으로 자릅니다. 로고가 PNG의 절대 가장자리까지 뻗어 있으면 모서리가 잘려 나갑니다.

  • 모든 중요한 시각적 요소는 이미지 내부의 80% 영역 내에 유지하세요.
  • 단색 배경을 사용하세요. 투명한 마스킹 가능 아이콘은 기기 테마에 따라 거친 흰색 또는 검은색 배경으로 렌더링됩니다.
  • Chrome DevTools의 Application > Manifest 탭을 사용하여 아이콘을 테스트하고 잘렸을 때 어떻게 보이는지 확인하세요.

캐시 무효화는 필수

파비콘은 인터넷에서 가장 공격적으로 캐시되는 자산입니다. 브라우저는 이를 두 번 다운로드하는 것을 매우 싫어합니다. 로고를 업데이트하고 서버의 파일만 교체한다면, 일반 사용자는 몇 달 동안 새 아이콘을 보지 못할 수 있습니다.

아이콘을 업데이트할 때는 항상 캐시 무효화 쿼리 문자열이나 파일 해시를 사용하세요. href='/icon.svg'를 href='/icon.svg?v=2'로 변경합니다. 이렇게 하면 브라우저가 즉시 새로운 자산을 가져오도록 강제합니다.

Guía Completa de Iconos Web: Tu Stack de 2026

Deja de generar 30 tamaños diferentes de PNG. Es una pérdida masiva de ancho de banda y tiempo de desarrollo. Los navegadores modernos se han unificado en torno a un estándar mucho más simple y potente. Solo necesitas cuatro archivos específicos para cubrir el 99% de todos los dispositivos y navegadores.

1. La Base Moderna (SVG)

El núcleo de tu estrategia de iconos ahora debería ser un favicon SVG. Los Gráficos Vectoriales Escalables son independientes de la resolución, lo que significa que un solo archivo se ve nítido en una laptop de gama baja y en una pantalla Retina 6K. Más importante aún, el SVG permite CSS incrustado.

¿Por qué importa el CSS incrustado? Por el modo oscuro. Puedes escribir una media query directamente dentro de tu archivo SVG para que tu icono cambie de color según el tema del sistema del usuario. Si tienes un logo negro, se volverá blanco automáticamente cuando un usuario cambie su navegador al modo oscuro. No se requiere JavaScript.

2. El Ecosistema Apple (Apple Touch Icon)

Los dispositivos Apple son conocidos por ignorar los manifiestos web estándar para los iconos de la pantalla de inicio. Si un usuario toca 'Agregar a la pantalla de inicio' en Safari, iOS busca un archivo muy específico: el Apple Touch Icon.

Necesitas un solo PNG de 180x180 píxeles. No uses un fondo transparente para este archivo. iOS rellenará automáticamente cualquier transparencia con negro, lo que generalmente arruina tu logo. Dale un color de fondo sólido y asegúrate de que tu logo esté centrado con suficiente margen.

3. El Mundo de Android y PWA (Web App Manifest)

Los dispositivos Android y las Aplicaciones Web Progresivas (PWA) dependen de un archivo site.webmanifest. Este es un archivo JSON simple que le dice al sistema operativo cómo debe comportarse tu aplicación web cuando se instala.

Dentro de este manifiesto, defines tus iconos. Necesitas exactamente dos tamaños PNG aquí: 192x192 y 512x512. La versión de 192px se usa para los iconos de la pantalla de inicio, mientras que la versión de 512px se usa para las pantallas de presentación cuando se carga la aplicación. También debes declararlos como 'maskable' (enmascarables) para que Android pueda darles forma de círculos, gotas o cuadrados redondeados dependiendo de las preferencias del lanzador del usuario.

4. El Respaldo Heredado (ICO)

Todavía necesitamos un archivo favicon.ico. ¿Por qué? Porque los navegadores antiguos, los lectores de RSS y algunos web scrapers empresariales todavía tienen codificadas las solicitudes a /favicon.ico. Si no lo tienes, los registros de tu servidor se llenarán de molestos errores 404.

Mantén este archivo pequeño. Un solo ICO de 32x32 píxeles es perfecto. Proporciona suficiente detalle para los sistemas heredados sin ralentizar la carga inicial de tu página.

Ejemplo del Mundo Real: Cómo lo Hace GitHub

Mira GitHub. Si abres GitHub en una pestaña del navegador y cambias el sistema operativo entre el modo claro y oscuro, el pequeño icono de Octocat cambia instantáneamente de negro a blanco. No usan JavaScript complejo para intercambiar los elementos del DOM.

Usan un solo favicon SVG con una media query prefers-color-scheme. Aquí tienes una versión simplificada de exactamente lo que hacen dentro de su SVG:

<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 100 100'>
  <style>
    path { fill: #24292f; }
    @media (prefers-color-scheme: dark) {
      path { fill: #ffffff; }
    }
  </style>
  <path d='...' />
</svg>

Esto es elegante, ligero y es manejado completamente por el motor de renderizado del navegador. Es el estándar de oro para el desarrollo web en 2026.

El Código: Uniendo Todo

Tienes tus cuatro archivos: icon.svg, apple-touch-icon.png, icon-192.png y icon-512.png (referenciados en tu manifiesto). ¿Cómo los implementas realmente en tu HTML?

Agrega este fragmento exacto al <head> de tu documento. El orden importa aquí. Los navegadores leen de arriba a abajo y generalmente usarán el último icono compatible que encuentren.

<!-- 1. El respaldo heredado (mantenlo en el directorio raíz) -->
<link rel='icon' href='/favicon.ico' sizes='32x32'>

<!-- 2. El SVG moderno -->
<link rel='icon' href='/icon.svg' type='image/svg+xml'>

<!-- 3. El Apple Touch Icon -->
<link rel='apple-touch-icon' href='/apple-touch-icon.png'>

<!-- 4. El Web Manifest -->
<link rel='manifest' href='/site.webmanifest'>

¿Notas lo que falta? No hay PNGs de 16x16. No hay etiquetas meta para los mosaicos de Windows Metro. Microsoft Edge ahora usa el Web App Manifest, haciendo que esas viejas etiquetas propietarias sean obsoletas. Finalmente hemos llegado a un punto donde menos código equivale a una mejor cobertura.

Mejores Prácticas que Probablemente Estás Ignorando

Incluso con los archivos correctos, los desarrolladores frecuentemente cometen errores de diseño e implementación. Aquí están las lecciones aprendidas a la fuerza tras auditar miles de sitios.

Cuidado con la Zona Segura Enmascarable

Cuando defines tus iconos de 192x192 y 512x512 en tu manifiesto, debes incluir "purpose": "any maskable". Sin embargo, Android recortará agresivamente tu icono para adaptarlo a diferentes formas. Si tu logo se extiende hasta el borde absoluto del PNG, las esquinas se cortarán.

  • Mantén todos los elementos visuales críticos dentro del 80% interior de la imagen.
  • Usa un color de fondo sólido. Los iconos enmascarables transparentes se renderizarán con un fondo blanco o negro áspero dependiendo del tema del dispositivo.
  • Prueba tu icono usando Chrome DevTools bajo la pestaña Application > Manifest para ver cómo se ve recortado.

Romper la Caché es Obligatorio

Los favicons son los activos almacenados en caché de manera más agresiva en Internet. Los navegadores odian descargarlos dos veces. Si actualizas tu logo y solo reemplazas el archivo en tu servidor, es posible que tus usuarios habituales no vean el nuevo icono durante meses.

Siempre usa cadenas de consulta para romper la caché o hashes de archivos cuando actualices tus iconos. Cambia href='/icon.svg' a href='/icon.svg?v=2'. Esto obliga al navegador a obtener el activo nuevo inmediatamente.
