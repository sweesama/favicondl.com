Cloudflare Pages 网站图标设置指南 (2026 版)

If you just deployed your first static site on Cloudflare Pages and saw that generic blue rocket icon in the browser tab, you know the pain. It screams "I didn't finish the setup." Unlike WordPress where you click a button in the dashboard, or Next.js where you drop a file in a specific folder, Cloudflare Pages requires a slightly different mindset in 2026. You are dealing with a pure static environment, and the server doesn't guess where your icon lives.

Many developers assume placing a file named favicon.ico in the root is enough. It works for some browsers, but it fails the modern web standards test. We need a robust stack that handles Apple devices, Android home screens, and dark mode. This guide cuts through the confusion and gives you the exact file structure to make your site look professional immediately.

Why the Default Setup Fails

Cloudflare Pages is incredibly fast because it serves static assets from the edge. However, it doesn't have a backend to dynamically generate icons or handle complex routing for missing files. If you rely on the implicit /favicon.ico request, you might get a 404 error in your logs, or worse, a broken image icon in Safari.

Look at how GitHub handles their static sites. They don't just upload one file. They serve a vector SVG for modern browsers, a high-resolution PNG for mobile devices, and a legacy ICO for older Windows systems. This multi-format approach ensures your brand looks sharp whether a user is on an iPhone 16 or a legacy corporate laptop. We will replicate this strategy for your Cloudflare project.

The 2026 File-Based Metadata Strategy

The most reliable way to set a favicon for Cloudflare Pages is to treat your public directory as the source of truth. Cloudflare Pages maps this folder directly to the root of your domain. There are no build steps required for the icons themselves, but the HTML referencing them must be correct.

Step 1: Prepare Your Icon Assets

Before touching the code, you need the right files. Do not just resize your logo. A complex logo looks like a muddy blob at 16x16 pixels. You need simplified versions. If you have a complex SVG logo, use a tool like Mzu favicondl to generate a simplified, high-contrast version specifically for the browser tab.

You need three specific files ready in your local project folder:

  • favicon.ico: The legacy container file. Yes, we still need this for some corporate proxies and older browsers.
  • icon.svg: The modern standard. Supports dark mode and scales perfectly to any size.
  • apple-touch-icon.png: A 180x180 pixel PNG for iOS home screens.

Step 2: Organize Your Directory Structure

Cloudflare Pages expects files to be in the root of your public folder (or whatever your build output directory is named). Do not nest your icons in a /images or /assets folder if you want the simplest setup. The browser expects the icon at the root URL.

Place your files exactly like this:

public/
  index.html
  favicon.ico
  icon.svg
  apple-touch-icon.png
  manifest.json

This structure ensures that when a browser requests https://your-site.pages.dev/favicon.ico, Cloudflare serves the file instantly without any 404 errors.

Step 3: Update Your HTML Head

Simply dropping files isn't enough. You must tell the browser which file to use for which context. Open your index.html and replace the default meta tags with this robust stack. This code handles SVG dark mode, Apple devices, and legacy fallbacks in one go.

<!-- Modern SVG with Dark Mode Support -->
<link rel="icon" type="image/svg+xml" href="/icon.svg">

<!-- Legacy Fallback for older browsers -->
<link rel="icon" type="image/x-icon" href="/favicon.ico">

<!-- Apple Touch Icon for iOS -->
<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">

<!-- Web App Manifest for Android/PWA -->
<link rel="manifest" href="/manifest.json">

Note the order here. We list the SVG first. Modern browsers will pick that up and ignore the rest. Older browsers will skip the SVG and grab the ICO. This is the most efficient way to handle compatibility without bloating your page load.

Common Pitfalls and How to Avoid Them

Even with the correct code, things can go wrong. The most common issue is caching. Cloudflare is aggressive with caching. If you update your icon but users still see the old one, it's not a bug in your code; it's the CDN.

The Cache Buster Trick

If you are deploying a new icon and it refuses to update, you need to force the browser to fetch the new version. Add a query string to your file paths. This is a standard practice for static sites.

<link rel="icon" type="image/svg+xml" href="/icon.svg?v=20260101">

Change the version number every time you update the image. This tricks the browser into thinking it's a new file, bypassing the Cloudflare edge cache immediately.

Check Your Build Process

If you are using a framework like React, Vue, or Astro, ensure your build process isn't deleting your icons. Some build tools clean the dist or build folder before deployment. Verify that your public folder contents are being copied to the final output directory. If you are using a custom build script, make sure it includes the copy command for your static assets.

Testing Your Implementation

Don't just trust your local preview. Deploy to a production URL and test it in an Incognito window. Open the browser console and look for network requests. You should see a 200 OK status for your icon.svg and favicon.ico. If you see 404s, check your file paths again.

For a deeper audit, use Mzu favicondl to scan your live URL. It will tell you exactly which icons are missing, if your dark mode SVG is valid, and if your Apple Touch Icon is the correct size. This saves hours of debugging on mobile devices.

Setting up a favicon for Cloudflare Pages doesn't have to be a mystery. By using a file-based approach with proper HTML tags, you ensure your site looks professional across every device. Stop serving the default rocket and start owning your brand identity in the browser tab today.

2026 年基于文件的元数据策略

在 Cloudflare Pages 上设置 favicon 最靠谱的方法，是把 public 目录视为唯一的真理来源。Cloudflare Pages 会直接将这个文件夹映射到你的域名根目录。图标本身不需要构建步骤，但引用它们的 HTML 代码必须精准无误。

第一步：准备图标资源

在碰代码之前，你得先搞定正确的文件。千万别只是简单缩放你的 Logo。复杂的 Logo 缩到 16x16 像素时，看起来就像一团模糊的墨迹。你需要简化版本。如果你的 SVG Logo 很复杂，建议用 Mzu favicondl 这类工具，专门生成一个高对比度、适合浏览器标签页显示的简化版。

  • icon.svg: 现代标准。支持深色模式，且能无损缩放到任意尺寸。
  • apple-touch-icon.png: 180x180 像素的 PNG，专供 iOS 主屏幕使用。

第二步：整理目录结构

Cloudflare Pages 期望文件位于 public 文件夹的根目录（或者是你的构建输出目录）。如果你想用最简单的配置，千万别把图标嵌套在 /images 或 /assets 文件夹里。浏览器默认就在根 URL 下寻找图标。

请严格按照以下结构放置文件：

第三步：更新 HTML Head

光把文件丢进去是不够的，你必须明确告诉浏览器在不同场景下该用哪个文件。打开你的 index.html，用下面这套健壮的代码替换默认的 meta 标签。这段代码一次性解决了 SVG 深色模式、Apple 设备适配以及旧版浏览器的回退方案。

<!-- 支持深色模式的现代 SVG -->
<link rel="icon" type="image/svg+xml" href="/icon.svg">

<!-- 旧版浏览器回退方案 -->
<link rel="icon" type="image/x-icon" href="/favicon.ico">

<!-- iOS 设备的主屏幕图标 -->
<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">

<!-- Android/PWA 的 Web App Manifest -->
<link rel="manifest" href="/manifest.json">

注意这里的顺序。我们把 SVG 放在最前面。现代浏览器会优先加载它并忽略后面的内容；旧版浏览器则会跳过 SVG，直接抓取 ICO。这是在保证兼容性的同时，避免页面加载臃肿的最优解。

常见陷阱与避坑指南

即使代码完全正确，问题也可能出在其他地方。最常见的问题就是缓存。Cloudflare 的缓存策略非常激进。如果你更新了图标但用户看到的还是旧的，别怀疑代码有 bug，那是 CDN 在作怪。

缓存清除技巧 (Cache Buster)

如果你部署了新图标却死活不更新，那就得强制浏览器去拉取新版本了。给文件路径加上查询字符串（Query String）。这是静态站点的标准操作。

<link rel="icon" type="image/svg+xml" href="/icon.svg?v=20260101">

每次更新图片时，修改版本号。这招能“骗”过浏览器，让它以为这是一个新文件，从而立即绕过 Cloudflare 的边缘缓存。

检查你的构建流程

如果你用的是 React、Vue 或 Astro 这类框架，务必确认构建流程没有误删你的图标。有些构建工具在部署前会清理 dist 或 build 文件夹。请验证你的 public 文件夹内容是否被正确复制到了最终的输出目录。如果你用的是自定义构建脚本，确保里面包含了复制静态资源的 copy 命令。

别只依赖本地预览。一定要部署到生产环境，用无痕模式测试。打开浏览器控制台，查看网络请求。你应该能看到 icon.svg 和 favicon.ico 返回 200 OK 状态。如果看到 404，请再次检查文件路径。

デフォルト設定が失敗する理由

Cloudflare Pages はエッジから静的アセットを提供するため非常に高速ですが、アイコンを動的に生成したり、欠落したファイルの複雑なルーティングを処理するバックエンドは備えていません。暗黙的な /favicon.ico リクエストに依存している場合、ログに 404 エラーが表示されるか、最悪の場合は Safari で「画像が破損」アイコンが表示されてしまいます。

GitHub が静的サイトをどのように扱っているかを見てみましょう。彼らは単一のファイルをアップロードするだけではありません。現代のブラウザ向けにベクター形式の SVG、モバイル端末向けに高解像度の PNG、そして古い Windows システム向けにレガシーな ICO を提供しています。このマルチフォーマットのアプローチにより、ユーザーが iPhone 16 を使っていようとも、古い法人向けラップトップを使っていようとも、ブランドが鮮明に見えるようにしています。この戦略をあなたの Cloudflare プロジェクトでも再現しましょう。

2026 年版：ファイルベースのメタデータ戦略

Cloudflare Pages 用の favicon を設定する最も信頼性の高い方法は、public ディレクトリを「真実の源（ソース・オブ・トゥルース）」として扱うことです。Cloudflare Pages はこのフォルダをドメインのルートに直接マッピングします。アイコン自体にビルドステップは不要ですが、それを参照する HTML が正しく記述されている必要があります。

ステップ 1：アイコンアセットの準備

コードに触れる前に、正しいファイルを用意しましょう。ロゴを単にリサイズするだけでは不十分です。複雑なロゴは 16x16 ピクセルでは汚い塊のように見えてしまいます。簡略化されたバージョンが必要です。複雑な SVG ロゴをお持ちの場合は、Mzu favicondl などのツールを使用して、ブラウザのタブ用に最適化された、高コントラストで簡略化されたバージョンを生成することをお勧めします。

ローカルのプロジェクトフォルダには、以下の 3 つのファイルを用意してください：

  • favicon.ico: レガシーなコンテナファイル。はい、一部の企業プロキシや古いブラウザのために、まだこれが必要です。
  • icon.svg: 現代の標準。ダークモードに対応し、あらゆるサイズに完璧にスケーリングします。
  • apple-touch-icon.png: iOS のホーム画面用の 180x180 ピクセルの PNG ファイル。

ステップ 2：ディレクトリ構造の整理

Cloudflare Pages は、ファイルが public フォルダのルート（またはビルド出力ディレクトリとして指定されているフォルダ）にあることを期待しています。最もシンプルな設定を実現したい場合は、アイコンを /images や /assets フォルダにネストしないでください。ブラウザはルートの URL にアイコンがあることを前提としています。

ファイルを以下のように配置してください：

public/
  index.html
  favicon.ico
  icon.svg
  apple-touch-icon.png
  manifest.json

この構造により、ブラウザが https://your-site.pages.dev/favicon.ico をリクエストした際、Cloudflare は 404 エラーなしで即座にファイルを配信できます。

ステップ 3：HTML の Head を更新する

ファイルを置くだけでは不十分です。ブラウザに、どのコンテキストでどのファイルを使用するかを指示する必要があります。index.html を開き、デフォルトのメタタグをこの堅牢なスタックに置き換えてください。このコードは、SVG のダークモード対応、Apple 端末、そしてレガシーなフォールバックを一度に処理します。

<!-- Modern SVG with Dark Mode Support -->
<link rel="icon" type="image/svg+xml" href="/icon.svg">

<!-- Legacy Fallback for older browsers -->
<link rel="icon" type="image/x-icon" href="/favicon.ico">

<!-- Apple Touch Icon for iOS -->
<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">

<!-- Web App Manifest for Android/PWA -->
<link rel="manifest" href="/manifest.json">

順序に注意してください。まず SVG をリストしています。現代のブラウザはそれを読み取り、残りを無視します。古いブラウザは SVG をスキップして ICO を取得します。これが、ページ読み込みを肥大化させずに互換性を処理する最も効率的な方法です。

一般的な落とし穴とその回避策

正しいコードを使用していても、問題が発生することがあります。最も一般的な問題はキャッシュです。Cloudflare のキャッシュは非常に積極的です。アイコンを更新してもユーザーが古いものを見ている場合、コードのバグではなく CDN の問題です。

キャッシュバスターのテクニック

新しいアイコンをデプロイしても更新されない場合、ブラウザに新しいバージョンを取得させる必要があります。ファイルパスにクエリ文字列を追加します。これは静的サイトにおける標準的なプラクティスです。

<link rel="icon" type="image/svg+xml" href="/icon.svg?v=20260101">

画像を更新するたびにバージョン番号を変更してください。これにより、ブラウザが新しいファイルだと認識し、即座に Cloudflare エッジキャッシュをバイパスできます。

ビルドプロセスの確認

React、Vue、Astro などのフレームワークを使用している場合、ビルドプロセスがアイコンを削除していないか確認してください。一部のビルドツールは、デプロイ前に dist や build フォルダをクリーンアップします。public フォルダの内容が最終出力ディレクトリにコピーされているか確認してください。カスタムビルドスクリプトを使用している場合は、静的アセットの copy コマンドが含まれていることを確認してください。

実装のテスト

ローカルのプレビューを信じるだけでは不十分です。本番 URL にデプロイし、シークレットウィンドウでテストしてください。ブラウザのコンソールを開き、ネットワークリクエストを確認します。icon.svg と favicon.ico に対して 200 OK ステータスが表示されるはずです。404 が表示される場合は、ファイルパスを再確認してください。

より詳細な監査を行うには、Mzu favicondl を使用してライブ URL をスキャンしてください。どのアイコンが欠落しているか、ダークモード用の SVG が有効か、Apple Touch Icon のサイズが正しいかを正確に教えてくれます。これにより、モバイルデバイスでのデバッグ時間を大幅に節約できます。

많은 개발자가 단순히 루트 디렉토리에 favicon.ico 파일 하나만 두면 된다고 생각합니다. 일부 브라우저에서는 작동할지 모르지만, 현대적인 웹 표준을 충족하지는 못합니다. 우리는 애플 기기, 안드로이드 홈 화면, 다크 모드까지 모두 커버할 수 있는 견고한 스택이 필요합니다. 이 가이드에서는 혼란을 정리하고, 사이트가 즉시 프로페셔널해 보일 수 있는 정확한 파일 구조를 제시합니다.

기본 설정이 실패하는 이유

클라우드플레어 페이지스는 에지 (Edge) 에서 정적 에셋을 제공하므로 incredibly 빠릅니다. 하지만 아이콘을 동적으로 생성하거나 누락된 파일을 위한 복잡한 라우팅을 처리할 백엔드가 없습니다. 암묵적인 /favicon.ico 요청에만 의존한다면, 로그에서 404 에러를 보게 되거나, 최악의 경우 사파리에서 깨진 이미지 아이콘이 표시될 수 있습니다.

깃허브가 정적 사이트를 어떻게 처리하는지 살펴보세요. 그들은 단순히 파일 하나를 올리지 않습니다. 현대 브라우저용 벡터 SVG, 모바일 기기용 고해상도 PNG, 구형 윈도우 시스템용 레거시 ICO를 각각 제공합니다. 이 멀티 포맷 접근 방식은 사용자가 아이폰 16 을 사용하든, 구형 기업용 노트북을 사용하든 브랜드가 선명하게 보이도록 보장합니다. 우리는 이 전략을 클라우드플레어 프로젝트에 그대로 적용할 것입니다.

2026 년 파일 기반 메타데이터 전략

클라우드플레어 페이지스에 favicon을 설정하는 가장 신뢰할 수 있는 방법은 public 디렉토리를 소스 오브 트루스 (Source of Truth) 로 간주하는 것입니다. 클라우드플레어 페이지스는 이 폴더를 도메인의 루트에 직접 매핑합니다. 아이콘 자체를 위한 빌드 스텝은 필요 없지만, 이를 참조하는 HTML 코드는 정확해야 합니다.

1 단계: 아이콘 에셋 준비하기

코드 작성 전에 올바른 파일이 필요합니다. 단순히 로고를 리사이즈해서는 안 됩니다. 복잡한 로고는 16x16 픽셀 크기로 축소되면 흐릿한 덩어리로 보입니다. 단순화된 버전이 필요합니다. 복잡한 SVG 로고를 사용 중이라면, 브라우저 탭에 표시될 수 있도록 대비가 높은 단순화된 버전을 생성하는 Mzu favicondl 같은 도구를 사용하세요.

로컬 프로젝트 폴더에 다음 세 가지 파일이 준비되어 있어야 합니다:

  • favicon.ico: 레거시 컨테이너 파일입니다. 네, 여전히 일부 기업용 프록시와 구형 브라우저를 위해 필요합니다.
  • icon.svg: 현대적인 표준입니다. 다크 모드를 지원하며 어떤 크기에서도 완벽하게 스케일링됩니다.
  • apple-touch-icon.png: iOS 홈 화면을 위한 180x180 픽셀 PNG 파일입니다.

2 단계: 디렉토리 구조 정리하기

클라우드플레어 페이지스는 파일이 public 폴더의 루트 (또는 빌드 출력 디렉토리 이름) 에 있기를 기대합니다. 가장 간단한 설정을 원한다면 아이콘을 /images나 /assets 폴더에 중첩하지 마세요. 브라우저는 루트 URL 에 아이콘이 있을 것으로 기대합니다.

파일을 정확히 다음과 같이 배치하세요:

public/
  index.html
  favicon.ico
  icon.svg
  apple-touch-icon.png
  manifest.json

이 구조를 통해 브라우저가 https://your-site.pages.dev/favicon.ico를 요청할 때, 클라우드플레어는 404 에러 없이 즉시 파일을 제공합니다.

3 단계: HTML 헤드 업데이트하기

단순히 파일을 넣는 것만으로는 부족합니다. 브라우저에게 어떤 컨텍스트에서 어떤 파일을 사용할지 알려줘야 합니다. index.html을 열고 기본 메타 태그를 이 견고한 스택으로 교체하세요. 이 코드는 SVG 다크 모드, 애플 기기, 레거시 폴백을 한 번에 처리합니다.

<!-- Modern SVG with Dark Mode Support -->
<link rel="icon" type="image/svg+xml" href="/icon.svg">

<!-- Legacy Fallback for older browsers -->
<link rel="icon" type="image/x-icon" href="/favicon.ico">

<!-- Apple Touch Icon for iOS -->
<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">

<!-- Web App Manifest for Android/PWA -->
<link rel="manifest" href="/manifest.json">

여기서 순서에 주의하세요. 우리는 SVG 를 먼저 나열합니다. 현대 브라우저는 이를 인식하고 나머지는 무시합니다. 구형 브라우저는 SVG 를 건너뛰고 ICO 를 가져옵니다. 이것이 페이지 로딩을 무겁게 하지 않으면서 호환성을 처리하는 가장 효율적인 방법입니다.

흔한 함정과 해결 방법

올바른 코드가 있더라도 문제가 발생할 수 있습니다. 가장 흔한 문제는 캐싱입니다. 클라우드플레어는 캐싱이 매우 공격적입니다. 아이콘을 업데이트했는데도 사용자가 여전히 이전 아이콘을 본다면, 코드에 버그가 있는 것이 아니라 CDN 때문입니다.

캐시 버스터 (Cache Buster) 트릭

새 아이콘을 배포했는데 업데이트가 안 된다면, 브라우저가 새 버전을 가져오도록 강제해야 합니다. 파일 경로에 쿼리 스트링을 추가하세요. 이는 정적 사이트에서 표준적인 관행입니다.

<link rel="icon" type="image/svg+xml" href="/icon.svg?v=20260101">

이미지를 업데이트할 때마다 버전 번호를 변경하세요. 이는 브라우저가 새로운 파일인 것처럼 착각하게 만들어 클라우드플레어 에지 캐시를 즉시 우회하게 합니다.

빌드 프로세스 확인하기

리액트, 뷰, 아스트로 같은 프레임워크를 사용 중이라면, 빌드 프로세스가 아이콘을 삭제하지 않는지 확인하세요. 일부 빌드 도구는 배포 전에 dist나 build 폴더를 정리합니다. public 폴더의 내용이 최종 출력 디렉토리로 복사되는지 확인하세요. 커스텀 빌드 스크립트를 사용 중이라면, 정적 에셋을 위한 copy 명령어가 포함되어 있는지 확인하세요.

구현 테스트하기

로컬 미리보기만 믿지 마세요. 프로덕션 URL 에 배포하고 시크릿 창에서 테스트하세요. 브라우저 콘솔을 열어 네트워크 요청을 확인하세요. icon.svg와 favicon.ico에 대해 200 OK 상태 코드가 표시되어야 합니다. 404 가 표시된다면 파일 경로를 다시 확인하세요.

La estrategia de metadatos basada en archivos para 2026

La forma más fiable de establecer un favicon para Cloudflare Pages es tratar tu directorio public como la fuente de la verdad. Cloudflare Pages mapea esta carpeta directamente a la raíz de tu dominio. No se requieren pasos de compilación para los propios iconos, pero el HTML que los referencia debe ser impecable.

Paso 1: Prepara tus activos de icono

Antes de tocar una sola línea de código, necesitas los archivos correctos. No te limites a redimensionar tu logotipo. Un logotipo complejo se ve como una mancha borrosa a 16x16 píxeles. Necesitas versiones simplificadas. Si tienes un logotipo SVG complejo, usa una herramienta como Mzu favicondl para generar una versión simplificada y de alto contraste específicamente para la pestaña del navegador.

Necesitas tener listos tres archivos específicos en tu carpeta de proyecto local:

  • favicon.ico: El contenedor legado. Sí, todavía lo necesitamos para algunos proxies corporativos y navegadores antiguos.
  • icon.svg: El estándar moderno. Soporta modo oscuro y escala perfectamente a cualquier tamaño.
  • apple-touch-icon.png: Un PNG de 180x180 píxeles para las pantallas de inicio de iOS.

Paso 2: Organiza tu estructura de directorios

Cloudflare Pages espera que los archivos estén en la raíz de tu carpeta public (o como se llame tu directorio de salida de compilación). No anides tus iconos en una carpeta /images o /assets si quieres la configuración más sencilla. El navegador espera encontrar el icono en la URL raíz.

Coloca tus archivos exactamente así:

public/
  index.html
  favicon.ico
  icon.svg
  apple-touch-icon.png
  manifest.json

Esta estructura asegura que, cuando un navegador solicite https://tu-sitio.pages.dev/favicon.ico, Cloudflare sirva el archivo instantáneamente sin generar errores 404.

Paso 3: Actualiza el head de tu HTML

Simplemente soltar los archivos no es suficiente. Debes decirle al navegador qué archivo usar para cada contexto. Abre tu index.html y reemplaza las etiquetas meta por defecto con este stack robusto. Este código maneja el modo oscuro en SVG, dispositivos Apple y fallbacks legados de una sola vez.

<!-- Modern SVG con soporte para modo oscuro -->
<link rel="icon" type="image/svg+xml" href="/icon.svg">

<!-- Fallback legado para navegadores antiguos -->
<link rel="icon" type="image/x-icon" href="/favicon.ico">

<!-- Apple Touch Icon para iOS -->
<link rel="apple-touch-icon" sizes="180x180" href="/apple-touch-icon.png">

<!-- Web App Manifest para Android/PWA -->
<link rel="manifest" href="/manifest.json">

Fíjate en el orden aquí. Listamos el SVG primero. Los navegadores modernos lo detectarán e ignorarán el resto. Los navegadores antiguos saltarán el SVG y tomarán el ICO. Esta es la forma más eficiente de manejar la compatibilidad sin inflar la carga de tu página.

Errores comunes y cómo evitarlos

Incluso con el código correcto, las cosas pueden salir mal. El problema más habitual es la caché. Cloudflare es agresivo con el caching. Si actualizas tu icono pero los usuarios siguen viendo el antiguo, no es un bug en tu código; es el CDN.

El truco del Cache Buster

Si estás desplegando un nuevo icono y se niega a actualizarse, necesitas forzar al navegador a obtener la nueva versión. Añade un parámetro de consulta (query string) a las rutas de tus archivos. Esta es una práctica estándar para sitios estáticos.

<link rel="icon" type="image/svg+xml" href="/icon.svg?v=20260101">

Cambia el número de versión cada vez que actualices la imagen. Esto engaña al navegador haciéndole pensar que es un archivo nuevo, omitiendo la caché edge de Cloudflare de inmediato.

Si estás usando un framework como React, Vue o Astro, asegúrate de que tu proceso de compilación no esté borrando tus iconos. Algunas herramientas de build limpian la carpeta dist o build antes del despliegue. Verifica que el contenido de tu carpeta public se esté copiando al directorio de salida final. Si usas un script de build personalizado, asegúrate de que incluya el comando copy para tus activos estáticos.
